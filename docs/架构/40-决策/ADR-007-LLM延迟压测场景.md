# ADR-007：压测补充 LLM 延迟场景

> **状态**：✅ 已决策
> **日期**：2026-06-06
> **作者**：第三方终审 → 系统架构组

---

## 背景

第三方终审发现 **C5 关注点**：当前千节点压测分析覆盖了内存、栈、CTE、DB 写入四个维度，但没有覆盖 LLM 调用的延迟。对于 CONDITIONAL 模式，每次阶段决策都要调用一次 LLM，1000 节点树最坏情况 = 1000 × LLM 延迟 ≈ 30+ 分钟。

架构评审（R03）独立发现了同一问题，并建议 LRU 缓存 + 5s 独立超时。

## 决策

在 `arc-09-千节点压测分析.md` §2.6 中追加 LLM 调用延迟的压测场景：

| 场景 | LLM 调用次数 | 单次延迟 | 总耗时 |
|------|-------------|---------|--------|
| CONDITIONAL（最坏情况，无缓存） | 1000 | 2s | > 30 min |
| CONDITIONAL + LRU 缓存（128 条目，10 分钟 TTL） | 100-200 | 2s | 3-7 min |

### 缓解措施

1. **LRU 缓存**（128 条目，10 分钟 TTL）：相同上下文重复到达时命中缓存，降低 LLM 调用 80-90%。
2. **独立超时**：每次 CONDITIONAL 决策设 5s 超时，超时后推进到下一阶段而非阻塞整棵树。
3. **压测验证项**：
   - LRU 缓存命中率（期望 ≥ 80%）
   - 缓存失效后的调用延迟
   - CONDITIONAL 模式 1000 节点端到端耗时

## 理由

1. LLM 调用是网络 IO，延迟不可控，压测必须覆盖此场景才能评估真实用户体验。
2. Phase 0 的 1000 节点压测作为架构验证，覆盖 CONDITIONAL 模式是必要条件。
3. 该场景同时验证 LRU 缓存的实效，为 Phase 1 的完整实现提供数据支撑。
