# ADR-016：审计日志与安全增强

> **状态**：✅ 已决策
> **日期**：2026-06-05
> **作者**：安全架构师初审 → 系统架构组

---

## 背景

初审（review-第三方评审报告.md）中安全架构师李工发现三项风险：

**R4：审计日志缺少 traceId。** `AuditManager.record()` 记录操作日志，但没有与可观测性的链路追踪关联。问题排查时，审计日志和 Trace 日志无法关联到同一次请求。

**R5：输出验证调用主 LLM 模型放大了延迟和费用。** `LLMAsJudgeValidator` 通过 `ModelRouter` 调用，复用主路径的模型路由。主路由可能路由到 GPT-4/top 模型，每次输出验证都产生一次高成本 + 高延迟的 LLM 调用。

**R6：MCP Preflight 检查缺少凭证有效性校验。** Preflight 检查工具是否存在、参数是否合规，但不验证 OAuth token 是否仍然有效。token 可能在 preflight 检查和实际调用之间被撤销。

## 决策

### R4：AuditManager 增加 traceId

```java
interface AuditManager {
    void record(String action, String resourceId, String accountId) {
        // 自动从 SpanContext 获取当前 traceId
        String traceId = SpanContext.currentTraceId();
        // 写入审计日志（含 traceId）
    }
}
```

### R5：LLMAsJudgeValidator 改用独立本地模型

```java
@Component
class LLMAsJudgeValidator implements OutputValidator {
    /** 独立适配器，不通过主 ModelRouter。 */
    private final OllamaAdapter judgeModel;
}
```

- 使用本地部署的 `OllamaAdapter`（如 llama3.2 8B）
- 不走 `ModelRouter.forTask()`，不占用主模型配额和预算
- 验证延迟从 2-5s（GPT-4）降为 0.5-2s（本地模型）

### R6：PreflightResult 增加凭证校验

```java
record PreflightResult(
    boolean toolsAvailable,
    boolean credentialValid,    // 新增：凭证是否有效
    List<String> availableTools,
    String detail
) {}

// 探针：使用 listTools（只读操作）验证凭证有效性
PreflightResult check(McpServerConfig config) {
    boolean credentialValid = mcpClient.listTools(config).isSuccess();
    // ...
}
```

`listTools` 是 MCP 协议的只读操作，无副作用，适合作为凭证探针。

## 理由

1. **R4**：traceId 是分布式系统中问题排查的锚点，审计日志必须包含 traceId 才能与链路追踪、业务日志关联。
2. **R5**：输出验证在主模型上做，延迟（2-5s）和费用（GPT-4 价格）不可接受。本地模型以略低准确率换取 10 倍以上的成本和延迟优化。
3. **R6**：OAuth token 随时可能被撤销（用户手动撤销、SSO session 过期），preflight 必须包含凭证验证才能反映真实可用状态。
