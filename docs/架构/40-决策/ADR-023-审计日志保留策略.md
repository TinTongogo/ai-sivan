# ADR-023：审计日志保留策略——90 天 TTL

> **状态**：✅ 已决策
> **日期**：2026-06-06
> **作者**：交叉评审 → 系统架构组

---

## 背景

交叉评审发现 **吴-R2 关注点**：审计日志表会无限增长。`AuditLog` 表按 100 用户 × 每天 1000 次操作 × 365 天 = 3650 万条/年。

## 决策

审计日志表设 **90 天 TTL**，过期自动删除。

### 技术设计

| 方案 | 选择 | 理由 |
|------|------|------|
| 清理方式 | `DELETE FROM audit_log WHERE timestamp < NOW() - INTERVAL '90 days'` | PostgreSQL 直接清理，无外部依赖 |
| 执行频率 | 每日一次，低峰期（凌晨 3 点） | 不影响白天业务 |
| 分区策略 | 当前单表即可（900 万条/90 天） | 如果增长超预期，后续可 time-based 分区 |

### 归档策略

- 需要长期保留的审计数据，由外部日志平台（ELK、Loki 等）拉取
- Sivan 自身不承担长期归档的存储和管理责任

## 理由

1. 900 万条/90 天是 PostgreSQL 单表的舒适区间，无需分区。
2. 超出 90 天的审计日志在企业环境中应由专业日志平台管理。
3. 简单的 TTL 策略避免了"永不清除"导致的生产事故。
