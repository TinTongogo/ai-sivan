## 3. 设计原则在各领域的具体映射

以下展示六条统一原则如何映射到每个领域的代码层面。

### 3.1 零 switch(nodeType) — 所有类型分派走 Registry

| # | 领域 | 如果没有 switch，原本的 switch 变成什么 |
|---|---|---|
| 01 | 森林架构 | `switch(nodeType)` → `LeafExecutorRegistry.forType(nodeType)` |
| 02 | 工具动态感知 | `switch(toolName)` → `ToolRegistry.lookup(toolName)` |
| 03 | 本能模板与正向反馈 | `switch(matchStrategy)` → `TreeMatcher.matchers.forEach(m -> m.match(task))` |
| 04 | 对话历史智能压缩与提取 | `switch(contentType)` → `CompressorRegistry.forType(contentType)` |
| 05 | 沙箱与安全 | `switch(actionType)` → `SandboxPolicyRegistry.forType(actionType)` |
| 06 | 提示词管理与统一人格 | `switch(promptKey)` → `PromptRegistry.render(key, params)` |
| 09 | 知识库 RAG | `switch(retrievalMode)` → `RetrievalRegistry.forMode(mode)` |
| 12 | 记忆闪现 | `switch(recallStrategy)` → `MemoryRegistry.forStrategy(context)` |
| 14 | 用户画像 | `switch(learnerType)` → `ProfileRegistry.forType(type)` |
| 15 | 前端交互 | `switch(nodeType)` → `NodeRendererRegistry.forType(nodeType)` |

**判断标准**：任何新增 x 类型只需要新增一个类、实现对应接口、注册到 Registry。不需要改分派逻辑。

### 3.2 零 if(delivery) — 输出过滤走 Decorator 链

| # | 领域 | 输出目标 | 如何组合 |
|---|---|---|---|
| 01 | 森林架构 | SSE / 语音 / 静默 | `SummarySink(MetricsSink(ErrorLogSink(SseSink)))` |
| 03 | 本能模板与正向反馈 | 反馈记录 | `AsyncLogSink(DedupSink(DbSink))` |
| 05 | 沙箱与安全 | 审计日志 | `AnonymizeSink(BatchSink(DbSink))` |
| 07 | API 契约 | SSE + WebSocket | `MetricsSink(SseSink)` + `MetricsSink(WsSink)` |
| 11 | 可观测性 | Metric + Log + Trace | `BatchSink(SamplingSink(MetricSink))` |
| 15 | 前端交互 | 树展开/折叠/语音汇报 | `SummaryRenderer(DetailRenderer)` |

**判断标准**：新增一个输出渠道，不需要改任何业务逻辑代码，只需要在 Sink 工厂中加一个装饰器。

### 3.3 接口隔离 — 一个接口一个变化方向

| # | 领域 | 反面教材（胖接口） | 正确拆分 |
|---|---|---|---|
| 01 | 森林架构 | `ForestNode`（树结构+执行+压缩+内容） | `TreeNode` / `ExecutableNode` / `CompressibleNode` / `ContentNode` |
| 02 | 工具感知 | `Tool`（发现+执行+权限+日志） | `ToolProvider` / `ToolExecutor` / `ToolPermission` / `ToolAuditor` |
| 05 | 沙箱与安全 | `Sandbox`（隔离+校验+审计+限制） | `SandboxIsolation` / `AccessValidator` / `AuditLogger` / `RateLimiter` |
| 06 | 提示词管理与统一人格 | `PromptManager`（获取+渲染+缓存+人格） | `PromptProvider` / `PromptRenderer` / `PromptCache` / `PersonaResolver` |
| 09 | 知识库 RAG | `KnowledgeBase`（索引+搜索+重排+注入） | `Indexer` / `Retriever` / `Reranker` / `ContextInjector` |
| 12 | 记忆闪现 | `MemorySystem`（存+取+衰减+注入） | `MemoryStore` / `RecallStrategy` / `DecayStrategy` / `FlashbackInjector` |

**判断标准**：如果一个接口有超过 3 个不相关的方法组，或需要超过 1 个实现类才能完整实现，说明应该拆分。

### 3.4 领域事件驱动 — 跨上下文通信走事件

| 事件 | 发布者 | 订阅者 | 所属领域 |
|---|---|---|---|
| `NodeStatusChanged` | ForestExecutor | ProgressAggregator / 审计日志 | 01 → 01, 11 |
| `GoalTreeCompleted` | ForestExecutor | ForestScheduler / TemplateService / TTS | 01 → 01, 03, 15 |
| `NodeExecutionFailed` | LeafExecutor | NotificationService | 01, 05 → 15 |
| `MilestoneCompleted` | ModeDispatcher | 前端阶段切换 / ProgressAggregator | 01 → 01, 15 |
| `ToolExecutionFinished` | ToolExecutor | TemplateFeedback / 审计日志 | 02 → 03, 11 |

### 3.6 accountId 显式传递 — 零 ThreadLocal

所有 Repository 接口必须显式声明 accountId 参数：

```java
// ✅ 正确：显式传递
interface ForestRepository {
    Forest findByForestIdAndAccount(UUID forestId, UUID accountId);
    void saveNode(ForestNode node, UUID accountId);
}

// ❌ 禁止：隐式依赖
interface ForestRepository {
    Forest findById(UUID forestId);
    // ↑ accountId 从哪来的？不明确
}
```

| 隐式依赖来源 | v1.0 的做法 | v2.0 的做法 |
|---|---|---|
| TokenContext | ThreadLocal 透传 | `ExecutionContext.accountId` 显式参数 |
| JWT 过滤器 | SecurityContextReactive 写入 attributes | `@RequestParam UUID accountId` |
| 事件订阅者 | 从发布事件的上下文推断 | 事件 record 中携带 `forestId` + `accountId` |
| 跨表查询 | JOIN 时隐式带 account_id | WHERE 子句显式 `account_id = ?` |

### 3.7 枢纽原则——不为用户增加负担

灵枢（Sivan）的含义是**枢纽**——连接用户与 AI 的通道，为用户创造价值，而不是让用户多做事。

每个功能在进入设计之前用这个标准检验：

```
枢纽（Pivot）：用户输入 → 灵枢处理 → 输出价值
                ↑
          这个环节不需要用户参与

负担（Burden）：用户输入 → 等待 → 用户手动操作 → 等待 → 输出价值
                          ↑
                   用户在这里多了一步
```

**决策规则**：

- 任何让用户多操作一步的功能，必须在另一处省回两步
- 默认关闭、用户主动开启，不等于"不增加负担"——开启操作本身就是负担
- 系统能自动做的事，不要让用户手动做
- 用户没想到但需要的信息，系统主动推送（Flashback）
- 用户不在意的中间过程，系统不展示（SUMMARY 模式）

## 5. 实现批次

### Batch 0：地基前的地基

**00-账号与项目管理**

所有其他领域的执行都依赖 `accountId` 和 `projectId`，必须先搭好账号/项目体系。其余领域的实现可以在此 Batch 完成前并行设计。

### Batch 1：地基

**01-森林架构** → **08-持久化恢复** → **05-沙箱安全** → **07-API 契约**

这四者相互依赖。森林架构定义数据结构，持久化恢复保证它不丢失，沙箱保证它安全，API 让前端接入。

### Batch 2：核心能力

**06-提示词管理** → **03-本能模板** → **02-工具动态感知** → **04-对话压缩**

提示词是 LeafExecutor 的 prompt 来源，本能模板是 TreeMatcher 的匹配源，工具是 TaskNode 的执行手段，压缩是 ForestCompressor 的消息来源。Batch 2 完成后，一个完整的 GoalTree 从创建、匹配、执行、到压缩回流的闭环才闭合。

### Batch 3：交互与数据

**09-知识库 RAG** + **12-记忆与闪现** + **13-对话管理** + **14-用户画像** + **15-前端交互**

依赖 Batch 2 的能力完备后，这些模块负责让系统有用（知识库 + 记忆）、可管理（对话）、个性化（画像）、好看（前端）。

### Batch 4：质量

**10-测试策略** + **11-可观测性** + **16-多模态** + **17-性能规模化**

贯穿全过程的纵向活动。多模态是独立能力、可随时接入。性能是约束条件而非功能。

## 6. 设计检查清单

每个领域的设计文档必须回答以下问题，否则不被批准进入实现阶段：

### 6.1 扩展性

- [ ] 是否有 `switch(nodeType)` / `if-else chain`？→ 全部改为 Registry
- [ ] 新增一个类型需要改几个文件？→ 只能 1 个（新增类），0 个被修改的类
- [ ] 扩展接口是否有 `default` 方法提供合理默认值？→ 新增者不需要实现全部方法
- [ ] 同一份分派逻辑是否出现在多个地方？→ 抽到 Registry

### 6.2 隔离性

- [ ] 稳定接口是否承载超过 3 个不相关的方法组？→ 拆分接口
- [ ] 胖接口是否被多处注入只为了用其中一个方法？→ 拆分接口
- [ ] 所有 Repository 方法是否显式带 `accountId`？→ 不允许 ThreadLocal
- [ ] 跨上下文调用是否直接依赖了仓储实现？→ 改为领域事件或调用方组装

### 6.3 输出

- [ ] 是否有 `if (delivery == STREAM/SUMMARY)` 之类的输出判断？→ 改为 Decorator
- [ ] 新增一个输出渠道需要改几个文件？→ 只能 1 个（新增 Sink），0 个被修改的类
- [ ] 业务逻辑代码中是否有 `logger.info` 之外的输出？→ 输出走 EventSink，不走 logger

### 6.4 事件

- [ ] 跨上下文的副作用是否通过领域事件触发？→ 不能直接调跨上下文的方法
- [ ] 事件 record 是否包含 `forestId` + `accountId`？→ 确保订阅者能定位所属用户
- [ ] 事件本身是否会触发级联事件风暴？→ 当前只有 3 个事件类型，控制增长
- [ ] 事件的发布和消费在同一个事务中？→ 保证一致性

### 6.5 模块边界

- [ ] 每个模块的包名是否独立？→ `com.icusu.sivan.{domain}`
- [ ] 模块之间的依赖方向是否与依赖拓扑一致？→ 不允许反向依赖
- [ ] 模块的 SPI（扩展接口）是否定义在模块的 API 包中？→ 实现类在 internal 包
