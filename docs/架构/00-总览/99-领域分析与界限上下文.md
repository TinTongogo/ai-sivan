# 领域分析与界限上下文

> 日期：2026-06-12

---

## 1. 问题域概述

Sivan 是一个"私人 AI 团队操作系统"。用户告诉它一个目标，它自主组建 Agent 团队、拆解任务、编排执行、交付结果。

```
用户输入 → 任务拆解 → Agent 分派 → 编排执行 → 结果汇总 → 交付用户
```

核心价值：**把"用户 vs 一个 AI"的对话模式升级为"用户 vs 一个 AI 团队"的协作模式。**

---

## 2. 界限上下文

### 2.1 核心域 (Core Domain)

#### Forest（森林）
目标树的全生命周期。这是 Sivan 最核心的领域。

| 聚合 | 职责 | 关键实体/值对象 |
|------|------|----------------|
| `Forest` | 一棵目标树的根，聚合根 | forestId, accountId, title, rootNodeId |
| `TreeNode` | 树节点 | nodeId, mode, status, metadata |
| `InnerGoalNode` | 内部节点，有编排模式 | mode, children |
| `TaskNode` | 叶子节点，可被 Agent 执行 | content, agentName |
| `SynthesisNode` | 汇总节点 | content - 收集子节点产出 |
| `NodeStatus` | 节点生命周期 | PENDING/RUNNING/COMPLETED/FAILED/CANCELLED/SKIPPED |
| `Mode` | 编排模式 | SEQUENTIAL/PARALLEL/CONDITIONAL/HIERARCHICAL/CONSENSUS |

**不变量**：
- 一棵森林有且只有一个根节点
- 内部节点必须有子节点
- 叶子节点必须有对应的 LeafExecutor
- 节点状态只能单向流转

#### Agent（智能体）
Agent 的定义、路由、技能绑定。

| 聚合 | 职责 | 关键实体/值对象 |
|------|------|----------------|
| `AgentDefinition` | Agent 配置 | agentId, name, systemPrompt, skillIds |
| `Skill` | 技能/工具绑定 | skillId, name, content |
| `RouteResult` | 路由决策结果 | agentName, tier, confidence |
| `ToolSpec` | 工具规范 | name, description, inputSchema |

**不变量**：
- 每个路由决策必须有 Agent 归属
- Skill 必须在 Agent 的 category 范围内

#### Orchestration（编排）
任务分解 + 树执行 + 结果汇总。

| 职责 | 说明 |
|------|------|
| `TreeMatcher` | 用户输入 → 任务树（本能模板匹配或 LLM 分解） |
| `ForestExecutor` | 遍历执行树，按 Mode 分派子节点 |
| `ModeStrategy` | 编排策略：顺序/并行/条件/层级/共识 |
| `LeafExecutor` | 叶子节点执行器 SPI |

---

### 2.2 支撑子域 (Supporting)

#### Conversation（对话）
对话是用户与 Sivan 的交互容器。Forest 执行结果通过 Conversation 交付。

| 聚合 | 职责 |
|------|------|
| `Conversation` | 会话容器，承载消息列表和关联知识库 |
| `Message` | 单条消息，含内容、附件、编排详情 |

**关联**：Conversation ←→ Forest（双向引用）

#### Tool Management（工具管理）
MCP 工具注册、发现、路由、调用。

| 聚合 | 职责 |
|------|------|
| `McpServerConfig` | MCP 服务器配置 |
| `ToolRegistry` | 工具注册中心（内部工具 + MCP 工具） |
| `ToolRouter` | 按叶子类型/任务路由工具 |

#### Memory（记忆）
三层记忆系统（SESSION → PROJECT → USER）

| 聚合 | 职责 |
|------|------|
| `Flashback` | 情境闪现推送 |
| `InstinctPattern` | 本能模板（历史成功的树结构） |
| `MemoryContext` | 记忆注入上下文 |

#### Knowledge Base（知识库）
文档索引、语义搜索、RAG。

| 聚合 | 职责 |
|------|------|
| `KnowledgeBase` | 知识库 |
| `KbDocument` | 知识库文档 |
| `Chunk` | 文档分块 |
| `SearchService` | 向量+全文+重排搜索 |

## 3. 当前分层问题

### 3.1 domain 层混杂了领域概念和执行协议

```
domain/forest/service/ （当前）
├─ ForestRepository        ← ✅ 领域仓储接口
├─ ProgressAggregator      ← ✅ 领域服务（进度计算）
├─ FeatureExtractor        ← ✅ 领域服务（特征提取）
├─ StructuralSimilarity    ← ✅ 领域服务（相似度计算）
├─ TreeMatcher             ← ✅ 领域服务（树匹配）
│
├─ LeafExecutor            ← ❌ SPI 接口（不包含业务规则）
├─ ModeStrategy            ← ❌ SPI 接口
├─ ModeDispatcher          ← ❌ SPI 接口
├─ Continuation            ← ❌ 回调接口
├─ EventSink               ← ❌ 事件发出口
├─ Span / SpanContext      ← ❌ 链路追踪
│
├─ ImageGenCapability      ← ❌ Provider SPI
├─ SpeechSynthCapability   ← ❌ Provider SPI
├─ SpeechRecogCapability   ← ❌ Provider SPI
├─ VideoGenCapability      ← ❌ Provider SPI
└─ ChatCapability          ← ❌ Provider SPI
```

### 3.2 core-api 定义混乱

```
core-api/core/
├─ agent/      ← Agent 协议（正确）
├─ model/      ← Model 协议（正确）
├─ tool/       ← Tool 协议（正确）
├─ message/    ← 消息类型（正确）
├─ context/    ← 执行上下文（正确）
│
└─ forest/     ← 刚搬进来的，需要重新评估
   ├─ ForestEvent      ← ✅ 事件协议
   ├─ EventSink        ← ✅ 事件发出口
   ├─ Span             ← ✅ 追踪数据
   ├─ SpanContext      ← ✅ 追踪上下文
   ├─ SpanExporter     ← ✅ 追踪导出
   ├─ ForestNode       ← ❓ 造出来迁就搬场的
   ├─ ForestContext    ← ❓ 同上
   ├─ LeafExecutor     ← ❓ 搬了但参数仍绑 domain
   ├─ Continuation     ← ❓ 同上
   ├─ ModeStrategy     ← ❓ 同上
   └─ ModeDispatcher   ← ❓ 同上
```

## 4. 修正方案

### 4.1 明确划分标准

一个接口该放哪里，不取决于"是不是接口"，而取决于**它的参数类型声明在哪个层**：

```
接口的参数/返回值全是 core-api 类型    → 放 core-api
接口的参数/返回值引用了 domain 类型    → 放 domain
```

### 4.2 最终分层

**core-api（协议层）** — 纯抽象，零业务规则

```
core/agent/            Agent 执行协议
core/model/            LLM 模型协议
core/tool/             工具调用协议  
core/message/          消息类型
core/forest/           森林执行事件协议
  ├─ ForestEvent       ← 事件 record，依赖 NodeStatus(common)
  ├─ EventSink         ← 事件发出口
  ├─ Span              ← 追踪 record
  ├─ SpanContext       ← 追踪上下文
  └─ SpanExporter      ← 追踪导出接口
```

**domain（领域层）** — 业务概念 + 依赖 core-api 的接口

```
domain/forest/tree/    树结构（领域核心）
domain/forest/service/ 领域服务 + SPI（参数涉及 TreeNode）
  ├─ LeafExecutor         ← 参数：TreeNode, ExecutionContext
  ├─ ModeStrategy         ← 参数：ForestNode
  ├─ ModeDispatcher       ← 参数：ForestNode
  ├─ Continuation         ← 参数：ForestNode
  ├─ ForestRepository     ← 仓储接口
  ├─ ProgressAggregator   ← 领域服务
  └─ ...

domain/forest/context/  ExecutionContext
domain/agent/           AgentDefinition, Skill
domain/tool/            McpServerConfig, ToolUsage
domain/conversation/    Conversation, Message
domain/knowledge/       KnowledgeBase, KbDocument
```

### 4.3 做与不做

**做**：
1. 把已搬到 core-api 的 `ForestNode`、`ForestContext` 删除（垃圾抽象）
2. 把 `LeafExecutor`、`Continuation`、`ModeStrategy`、`ModeDispatcher` 退回 domain
3. 保留 `ForestEvent`、`EventSink`、`Span` 系列在 core-api

**不做**：
- 不搬 `ImageGenCapability` 等 Provider SPI（参数涉及 domain 的 `ModelParams`）
- 不拆 `ExecutionContext`（它是 domain 概念，Field 含 `accountId` `projectId` 等业务字段）
- 不新增影子接口
