## 2. 深树递归

```java
/**
 * 方案：递归不设深度限制，但每个节点执行在独立调度上，
 * 避免栈帧累积。
 */
private Flux<OrchestrationEvent> executeNode(ExecutableNode node, ExecutionContext ctx, int depth) {
    // 深度保护
    if (depth > 500) {
        log.warn("递归深度超限: depth={}, nodeId={}", depth, node.nodeId());
        node.setStatus(SKIPPED);
        persistNode(node);
        return Flux.empty();
    }

    // 深度 > 50 时：每个子树在一个独立调度上运行，释放栈空间
    if (depth > 50) {
        return Flux.deferContextual(cv ->
            executeNodeInternal(node, ctx, depth)
        ).subscribeOn(Schedulers.boundedElastic());
    }

    return executeNodeInternal(node, ctx, depth);
}
```

---

#### boundedElastic 的适用性

深度 > 50 时切换到 `Schedulers.boundedElastic()` 有两个已知风险：

1. **上下文丢失**：`boundedElastic` 是阻塞任务调度器，Reactor 的 `Context` 在 `publishOn`/`subscribeOn` 切换时可能丢失（`Hooks.enableAutomaticContextPropagation()` 可缓解但非 100% 可靠）。
2. **线程膨胀**：`boundedElastic` 默认最大线程数 = 10 × CPU 核数。100 个并发深树遍历可能创建数百个线程。

**替代方案**：深树遍历优先使用 `Schedulers.parallel()`（固定线程数 = CPU 核数，适合非阻塞计算）。仅在确实需要阻塞调用（如 LLM 请求的同步等待）时才使用 `boundedElastic`。切换时显式传递 `Context`：

```java
// 推荐：显式传递 Context
Mono.defer(() -> blockingCall())
    .subscribeOn(Schedulers.boundedElastic())
    .contextWrite(ctx);  // 携带上游 Context
```

## 3. CTE 优化

| 措施 | 效果 |
|---|---|
| `WITH RECURSIVE` + `UNION ALL` | 避免排序，逐层追加 |
| 索引 `(forest_id, parent_id)` | 递归 JOIN 走索引 |
| 限制最大深度 `1000` | 防止无限递归 |
| 只查询可执行叶子类型 | `WHERE node_type IN ('task', 'home_task', 'synthesis')` |

```sql
-- 优化后的进度查询
WITH RECURSIVE subtree AS (
    SELECT node_id, parent_id, node_type, status, 0 AS depth
    FROM forest_nodes
    WHERE node_id = :rootId
  UNION ALL
    SELECT n.node_id, n.parent_id, n.node_type, n.status, s.depth + 1
    FROM forest_nodes n
    JOIN subtree s ON n.parent_id = s.node_id
    WHERE s.depth < 1000  -- 保护
)
SELECT
    COUNT(*) FILTER (WHERE node_type IN ('task','home_task','synthesis')) AS total,
    COUNT(*) FILTER (WHERE status = 'COMPLETED') AS completed
FROM subtree;
```

## 5. 性能预算与余量

编排引擎的性能需求远低于 LLM 调用的延迟。以下测算展示当前架构的潜力。

| 环节 | 基准值 | 瓶颈类型 | 上限 |
|---|---|---|---|
| 1000 节点树内存 | ~200KB | Java 对象头对齐 | ~150KB（GraalVM AOT） |
| 单节点分发耗时 | ~0.01ms | 虚方法分派（C2 去虚拟化后归零） | ~0.005ms |
| 1000 节点串行执行 | ~10ms | Flux.concatWith 内部对象分配 | ~5ms |
| CTE 1000 节点查询 | 2-5ms | PostgreSQL 递归执行计划 | 1-3ms |
| LLM 单次调用 | 1000-2000ms | 网络 I/O + 模型推理 | — |

**编排引擎在 LLM 延迟面前处于"空闲"状态。** 1000 节点分派耗时 ~10ms，而一次 LLM 调用至少 1000ms。编排引擎在等 LLM 的时间里可以跑完 10 万次节点分派。**性能瓶颈不在代码执行速度上，在 LLM 延迟上。**

### 5.1 ZGC 暂停保障

| GC 配置 | 最大暂停 |
|---|---|
| ZGC: `-XX:+UseZGC -Xmx2g` | **< 1ms** |
| ZGC: `-XX:+UseZGC -Xmx512m` | **< 1ms** |
| G1GC（默认） | 5-50ms |

1000 节点树仅 ~200KB，512MB 堆绰绰有余。

### 5.2 metadata 内存开销

文档中 10000 节点 < 500MB 的估算未包含 `metadata`（`Map<String, Object>`）的开销。每个节点的 metadata 存储 5 个键值对时的额外内存：

| 场景 | 单节点 metadata | 10000 节点 |
|------|----------------|-----------|
| 空 metadata | ~72 bytes (空 HashMap) | ~0.7 MB |
| 5 个短键值对（如 hitl=true） | ~400 bytes | ~4 MB |
| 5 个长键值对（如 agent_name=code_review_v3） | ~1 KB | ~10 MB |
| 含嵌套对象（如 condition 规则） | ~5 KB | ~50 MB |

**结论**：metadata 内存开销在典型场景下在可接受范围内（10000 节点 < 50 MB），但应避免在 metadata 中存储大数据对象（如完整文档内容、Base64 编码的图片）。大数据应存储为独立实体，metadata 中仅保留引用 key。

**约束**：每个节点的 metadata 键值对不超过 10 个，单个 value 不超过 1 KB。超出此范围需评审。

### 5.3 GraalVM AOT 选项

Java 21 + GraalVM Native Image 可将启动时间从 ~2s 降至 < 100ms，内存进一步压缩。零代码改动，仅需一个 `native-image` 构建步骤。

### 5.3 语言替代分析

如果未来出现 LLM 调用占比下降、高频实时交互场景：

| 方案 | 收益 | 成本 |
|---|---|---|
| Java 21 + ZGC（当前） | — | 0 |
| GraalVM Native Image | 启动 < 100ms，内存 -25% | 1 天配置 |
| Kotlin + GraalVM | 渐进替代，同一 JVM | 逐步替换 |
| Go | 内存 -30%，并发更直观 | 3 个月重写 |
| Rust | 内存 -75%，零 GC | 6 个月重写 |

换语言解决的是 JVM 那 ~1% 的性能问题。当前阶段 LLM 调用占 95% 以上的耗时，换语言全局收益低于 2-3%。

### 5.4 连接池配置

```yaml
sivan:
  datasource:
    hikari:
      maximum-pool-size: 50
      minimum-idle: 10
      connection-timeout: 5000
      max-lifetime: 600000
```

10 个并发 GoalTree × 5 个 PARALLEL TaskNode = 50 个并发线程，每个线程持有一个 DB 连接。50 的连接池可覆盖此需求。
