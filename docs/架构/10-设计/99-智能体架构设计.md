# 智能体架构设计

> 日期：2026-06-12

---

## 1. 设计目标

### 1.1 核心能力

| 能力 | 说明 |
|------|------|
| 进化学习 | 系统从每次交互中学习，逐步提高编排质量、降低 token 消耗 |
| 抗降智 | 阻止用户通过反复使用低效编排来"训练"系统变笨 |
| 渐进优化 | 新用户即用即好，老用户越用越聪明 |
| 透明可解释 | 每一次编排出处可追溯，为什么这样组织对用户可见 |

### 1.2 威胁模型

| 攻击方式 | 示例 | 影响 |
|---------|------|------|
| 重复低效 | 反复用 5 个 Agent 解决"查天气" | 系统学会用复杂流程处理简单任务 |
| 偏好污染 | 坚持用过时的技术栈 | Profile 学习偏向劣质选择 |
| 负反馈滥用 | 对正确结果点踩，对错误结果点赞 | 路由反馈信号混乱 |
| 模式固化 | 长期重复同一路径，抑制探索 | 系统无法发现更优解 |

## 3. L1 感知层

### 3.1 复杂度评估

防止降智的第一道防线：**准确判断任务的真实复杂度**。

```java
interface ComplexityScorer {
    TaskComplexity estimate(String input, UserProfile profile);

    record TaskComplexity(
        int score,           // 1-10
        String category,     // "simple_query" | "code_gen" | "analysis" | "creative"
        String[] dimensions, // ["scope", "ambiguity", "technical_depth"]
        int[] scores         // 各维度分数
    ) {
        static TaskComplexity simple() {
            return new TaskComplexity(1, "simple_query", new String[0], new int[0]);
        }
    }
}
```

**计算方法**（多信号融合）：

| 信号 | 权重 | 示例 |
|------|------|------|
| 输入长度 | 0.1 | <50字=1, >500字=6 |
| 关键词密度 | 0.2 | "帮我"=1, "重构"=5, "架构设计"=8 |
| 历史模式匹配 | 0.3 | 命中简单模式→调低，不命中→调高 |
| 用户画像 expertise | 0.2 | 专家用户的简单问题可能是真简单 |
| 模糊度 | 0.2 | 是否有明确约束条件 |

### 3.2 新奇度检测

判断当前任务与模式库中已有模式的**语义距离**。

```
高相似度(>0.9) → 直接使用已有模式
中相似度(0.6-0.9) → 变异已有模式
低相似度(<0.6) → 触发全量策略生成
```

**语义距离计算**：embedding 向量 + 结构特征(意图类型、复杂度维度)加权。

## 4. L2 策略层

### 4.1 模式库

模式库是系统的"经验"。每条模式包含：

```java
record ExecutionPattern(
    String patternId,
    String taskFingerprint,      // 任务语义指纹
    TaskComplexity complexity,   // 适用复杂度
    int templateVersion,        // 方案版本号

    // 编排方案
    ExecutableNode executionTree,

    // 效率得分（核心抗降智指标）
    EfficiencyScore efficiency,

    // 置信度
    Confidence confidence,

    // 对抗防御
    AntiAdversarialMeta antiAdverse,

    // 元数据
    Instant createdAt,
    Instant lastUsedAt,
    int totalUses,
    int successCount
) {
    boolean isTrustworthy() {
        return confidence.score() > 0.7
            && efficiency.wasteRatio() < 0.3
            && antiAdverse.generationCount() >= 3;
    }
}

record EfficiencyScore(
    double avgTokens,           // 平均 token 消耗
    double avgExecutionMs,      // 平均执行耗时
    double avgAgentCount,       // 平均使用 Agent 数
    double wasteRatio           // 资源浪费率 = 实际 / 基线
);

record Confidence(
    double score,                // 0-1
    int sampleSize,             // 采样量
    Instant lastUpdated,
    int crossUserConfirmations  // 跨用户确认次数
);
```

### 4.2 多样性调度器

即使有可信模式，也要定期探索新路径——这是进化学习的核心。

```java
interface DiversityScheduler {
    /**
     * 是否应该探索新方案而不是复用已有模式？
     */
    boolean shouldExplore(TaskComplexity complexity, ExecutionPattern existing);

    /**
     * 当前探索率（随时间衰减，但不会降到 0）
     */
    double currentExploreRate();
}
```

**探索策略**：

| 条件 | 探索率 | 理由 |
|------|--------|------|
| 新用户，<10 次交互 | 80% | 无足够数据，全面探索 |
| 有模式但置信度低 | 50% | 模式可能不可靠 |
| 有模式且置信度高 | 10% | 模式可信，但仍需验证 |
| 模式长期未更新 | 30% | 可能已有更优方案 |
| 复杂度与模式不匹配 | 60% | 怀疑模式被误用 |

**关键机制**：即使 `shouldExplore()` 返回 false，也有 **5% 的硬编码随机探索率**，永远不 100% 固化。

### 4.3 模式变异

新方案不是每次都从零生成，而是在已有模式上变异：

```
变异操作：
  ├─ 替换子节点 (Agent 替换)
  ├─ 合并阶段 (减少冗余)
  ├─ 拆分阶段 (细化粒度)
  ├─ 修改编排模式 (顺序→并行)
  └─ 工具精简 (移除未使用的工具)
```

变异后的方案与原始方案做 A/B 对比，保留更优者。

## 5. L3 执行层

### 5.1 质量监控

每次执行记录三个维度的数据：

```java
record ExecutionQuality(
    // 结果质量
    boolean userSatisfied,     // 用户显式反馈
    boolean taskCompleted,     // 是否完成
    String completionStatus,   // "success" | "partial" | "failed"

    // 效率质量
    long tokensUsed,
    long executionMs,
    int agentsUsed,
    int toolCallsMade,

    // 效率基线（关键抗降智指标）
    EfficiencyBaseline baseline
) {
    double efficiencyRatio() {
        return baseline.expectedTokens() > 0
            ? (double) tokensUsed / baseline.expectedTokens()
            : 1.0;
    }

    /** 是否触发降智告警阈值。效率比 > 3 即异常。 */
    boolean isAdversarialCandidate() {
        return efficiencyRatio() > 3.0;
    }
}

record EfficiencyBaseline(
    long expectedTokens,          // 同类任务预期 token
    long expectedExecutionMs,     // 同类任务预期耗时
    String baselineSource         // "historical" | "cross_user" | "theoretical"
);
```

### 5.2 基线计算

效率基线的准确度决定了抗降智能力。计算方法：

```java
class BaselineCalculator {
    EfficiencyBaseline calculate(TaskComplexity complexity) {
        // 1. 历史平均值：同类复杂度的历史执行数据
        HistoricalStats historical = loadHistoricalStats(complexity);

        // 2. 跨用户基准：其他用户处理同类问题的效率
        CrossUserStats crossUser = loadCrossUserStats(complexity);

        // 3. 理论下限：最优路径的理论最小值
        TheoreticalMin theoretical = estimateTheoreticalMin(complexity);

        // 取三者加权
        long expected = (long) (
            historical.avgTokens() * 0.4 +
            crossUser.p50Tokens() * 0.4 +
            theoretical.minTokens() * 0.2
        );

        return new EfficiencyBaseline(expected, ...);
    }
}
```

**关键**：跨用户基准阻止了单用户的偏好污染。一个人可以把自己的效率比拖垮，但不能影响其他人的基准。

## 6. L4 学习层

### 6.1 模式挖掘

从成功执行的树结构中提取可复用的模式。

```java
class PatternMiner {
    ExecutionPattern extract(ExecutableNode tree, ExecutionQuality quality) {
        // 1. 过滤：只保留质量合格的执行
        if (quality.isAdversarialCandidate()) {
            logAdversarialCandidate(tree, quality);
            return null;  // 不学习降智方案
        }

        // 2. 泛化：将具体 Agent 名替换为 Agent 类型
        ExecutableNode generalized = generalize(tree);

        // 3. 计算效率得分
        EfficiencyScore efficiency = scoreEfficiency(quality);

        // 4. 计算置信度（新模式初始置信度 = 0.3）
        Confidence confidence = new Confidence(0.3, 1, ...);

        return new ExecutionPattern(..., efficiency, confidence, ...);
    }
}
```

### 6.2 置信度-效率评分

这是核心抗降智设计——**评价标准不是"是否成功"，而是"是否高效成功"**。

| 场景 | 成功率 | 效率比 | 置信度变化 | 模式库操作 |
|------|--------|--------|-----------|-----------|
| 高效成功 | 1.0 | 0.8 (优于基线) | +0.15 | 提升排名 |
| 正常成功 | 1.0 | 1.0-2.0 | +0.05 | 保留 |
| 低效成功 | 1.0 | 3.0 (3倍资源) | -0.10 | **降权** |
| 低效失败 | 0 | 3.0 | -0.20 | **标记劣质** |
| 高效失败 | 0 | 0.8 | -0.05 | 环境问题，保留 |

**注意**：低效成功的置信度**反而降低**。这是抗降智的关键——用户不能通过反复让系统用低效方案完成任务来"教会"系统低效。

### 6.3 异常检测

识别潜在的降智攻击。

```java
class AdversarialDetector {
    AdversarialVerdict analyze(ExecutableNode tree, ExecutionQuality quality) {
        List<Signal> signals = new ArrayList<>();

        // 信号 1：复杂度-资源不匹配
        signals.add(complexityResourceMismatch(tree, quality));

        // 信号 2：同一用户的模式偏移
        signals.add(patternDrift(quality));

        // 信号 3：重复使用低效路径
        signals.add(repeatedInefficiency(quality));

        // 信号 4：反馈与质量矛盾
        signals.add(feedbackQualityConflict(quality));

        return aggregate(signals);
    }
}
```

**检测到降智后**：
1. 打印详细告警日志（用户、模式、效率比）
2. 该用户的模式对**跨用户基准**的贡献权重降低
3. 该用户的反馈信号权重降低（需要更多正常交互恢复）
4. 触发人工审查标志

### 6.4 元评估

跨用户、跨周期地评估模式库的整体健康度。

```java
class MetaEvaluator {
    void evaluate(PatternLibrary library) {
        for (ExecutionPattern pattern : library.allPatterns()) {
            // 1. 本周期淘汰率检查
            if (pattern.lastUsedAt().isBefore(now.minusDays(30))) {
                library.archive(pattern);  // 30 天未使用→归档
            }

            // 2. 效率退化检查
            if (pattern.efficiency().wasteRatio() > 0.5) {
                library.relegate(pattern); // 效率退化→降级
            }

            // 3. 跨用户不一致检查
            double crossUserScore = crossUserValidate(pattern);
            if (crossUserScore < 0.3) {
                library.markSuspect(pattern); // 跨用户不验证→标记
            }
        }
    }
}
```

### 6.5 剪枝引擎

定期清理模式库中的低质量模式。

```
剪枝触发条件：
  每 1000 次执行 或 每 24 小时

剪枝规则：
  ├─ 置信度 < 0.2 且 采样量 < 5    → 删除
  ├─ 效率比 > 5.0                   → 删除（极端低效）
  ├─ 30天未使用                      → 归档
  ├─ 有更新版本（更高效率）          → 删除旧版
  └─ 跨用户验证失败 > 3次            → 标记人工审核
```

## 7. 抗降智能力全景

| 防御层 | 机制 | 对抗的攻击 |
|--------|------|-----------|
| L1 复杂度评估 | 准确判断任务复杂度 | 用复杂方案解简单任务 |
| L2 多样性调度 | 强制探索新路径 | 路径固化 |
| L3 效率基线 | 跨用户基准 | 单用户偏好污染 |
| L3 质量监控 | 效率比告警 | 低效成功 |
| L4 模式挖掘过滤 | 拒绝降智方案入库 | 模式污染 |
| L4 置信度-效率评分 | 低效成功→降权 | 反馈滥用 |
| L4 异常检测 | 多信号综合判断 | 定向微调 |
| L4 元评估 | 跨用户验证 | 群体偏好偏移 |
| L4 剪枝引擎 | 定期清理 | 模式库劣化 |

## 8. 与 V2 关键差异

| 维度 | V2 | 新架构 |
|------|----|--------|
| 学习信号 | 成功/失败 | 成功+效率 |
| 置信度 | 使用次数 | 使用次数×效率比×跨用户确认 |
| 模式入库 | 直接存 | 通过 3 道过滤(效率/异常/泛化) |
| 探索率 | 无，命中即用 | 10% 基础探索率 |
| 跨用户保护 | 独立 | 跨用户基准+交叉验证 |
| 攻击检测 | 无 | 多信号异常检测 |
| 反馈权重 | 固定 | 根据用户可信度动态调整 |
