# 编排产出展示方案

> 日期：2026-06-12
> 状态：已实施

---

## 1. 问题定义

### 现状

Forest 编排模式下，多个 Agent 的产出通过 `accumulatedContext` 汇聚进一条 assistant 消息，前端看到的是混合文本，用户无法区分哪些是中间过程、哪些是最终产出，以及各 Agent 分别做了啥。

### 约束

1. **对话要保持线性** — 不能把每条 Agent 产出都变成独立消息气泡
2. **树要保持可遍历** — 用户随时可以钻入某阶段查看详情
3. **不重构消息模型** — 复用已有的消息结构和 SSE 协议

---

## 2. 方案：两层信息深度

```
Layer 1 — 对话面（默认）
┌─ 最终回复 ──────────────────────────────────┐
│ ✅ 登录模块已完成，包含：                       │
│ • 5 个 API 端点 (§架构师)                     │
│ • 输入校验 (§安全) [1 个问题 → 查看]           │
│ • 单元测试覆盖率 92% (§开发)                   │
│                                              │
│ [📋 查看执行详情 →]              耗时 12s · 3轮 │
└──────────────────────────────────────────────┘

Layer 2 — 执行树弹窗
└─ PipelineDialog → ForestTree 完整树（已有实现）
```

| 层级 | 展示方式 | 内容 |
|------|---------|------|
| Layer 1 | 消息气泡最终文本 | 含 `(§Agent)` 内联标注的最终回复 |
| Layer 2 | PipelineDialog 弹窗 | ForestTree 递归树，含节点状态/模式/耗时/Agent 归属 |

---

## 3. 实现

### 3.1 最终回复文本的内联标注

synthesis Agent 生成的最终回复中，对关键产出添加轻量归属标记 `(§Agent名)`：

```
"我们采用了 RESTful 风格 (§架构师)，做了输入校验 (§安全审查)"
```

前端 `ChatBubble.renderedContent` 通过 regex 替换渲染：

```
(§架构师) → <span class="agent-badge">🔍 架构师</span>
```

hover 时高亮，用户可感知"谁做了什么"。

### 3.2 执行树入口

消息底部元信息栏始终显示 `[📋 查看执行详情]` 按钮，点击打开 PipelineDialog，展示 ForestTree 完整执行树（含节点状态、模式标签、HITL 审批、A2A 消息）。

---

## 4. 不采用的方案（以及原因）

| 方案 | 放弃原因 |
|------|---------|
| 阶段折叠面板（PhaseCard） | 与执行树功能重叠；同一份 Forest 数据两份展示入口用户困惑 |
| 时间线条（timeline） | 同上 |
| 消息分拆多条气泡 | 违反约束 1，对话层会被冲乱 |
| 后端 sections JSONB 预处理 | 冗余——树本身就是完整的数据源，无需额外投影 |

---

## 5. 设计检查清单

- [ ] 对话面保持简洁？→ 是，Layer 1 约等于一条普通文本消息
- [ ] 用户可追溯各 Agent 产出？→ 是，`§` 标注 + PipelineDialog 展示树节点的 agentName
- [ ] 与现有 ForestTree 组件共存？→ 是，PipelineDialog 复用已有实现
- [ ] 不修改 SSE 协议？→ 是，仅在 ChatBubble 消费侧增强
- [ ] 不新增 DB 字段？→ 是，依赖已有的森林节点数据
