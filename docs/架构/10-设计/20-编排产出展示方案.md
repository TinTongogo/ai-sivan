# 编排产出展示方案 — 三层信息深度

> 日期：2026-06-12
> 状态：设计草案

---

## 1. 问题定义

### 现状

Forest 编排模式下，多个 Agent 的产出通过 `accumulatedContext` 汇聚进一条 assistant 消息，前端看到的是混合文本：

```
[用户] "帮我写一个登录模块"
→ [AI] "开始分析需求... (研究员输出) 正在设计API... (架构师输出)
   正在编码... (开发输出) 审查发现问题... (审查输出) 最终方案..."
```

用户无法区分：
- 哪些是中间过程、哪些是最终产出
- 当前执行到哪一步
- 各 Agent 分别做了啥

### 约束

1. **对话要保持线性** — 不能把每条 Agent 产出都变成独立消息气泡（对话会爆炸）
2. **树要保持可遍历** — 用户随时可以钻入某阶段查看详情
3. **与现有代码兼容** — 不重构消息模型，不改变 SSE 协议
4. **信息分层** — 默认简洁，按需深入

## 2. 方案：三层信息深度

```
Layer 1 — 对话面（默认）
┌─ 最终回复 ──────────────────────────────────┐
│ ✅ 登录模块已完成，包含：                       │
│ • 5 个 API 端点 (§架构师)                     │
│ • 输入校验 (§安全) [1 个问题 → 查看]           │
│ • 单元测试覆盖率 92% (§开发)                   │
│                                              │
│ 执行时间线 ▸ 分析 · 设计 · 编码 · 审查 · 完成   │
│ [📋 查看执行详情 →]              耗时 12s · 3轮 │
└──────────────────────────────────────────────┘

Layer 2 — 阶段折叠面板（点开查看各阶段）
┌─ 编排详情 ──────────────────────────────────┐
│ ✅ 需求分析     §研究员  3.2s  ──■── [展开 ▾] │
│    └─ 需求文档、技术选型                     │
│ ✅ API 设计     §架构师  4.1s  ──■── [展开 ▾] │
│    └─ OpenAPI 规范、数据模型                 │
│ ✅ 编码实现     §开发    5.5s  ──■── [展开 ▾] │
│    └─ 代码、测试                             │
│ ❌ 安全审查     §审查    2.0s  ──■── [展开 ▾] │
│    └─ 发现 SQL 注入风险                      │
└──────────────────────────────────────────────┘

Layer 3 — ForestTree 递归树（现有实现）
└─ 通过 "查看执行详情" 进入 PipelineDialog
```

## 3. 数据结构

### 3.1 编排阶段结构（message.sections）

利用已有 `messages.sections` JSONB 字段，每个阶段的结构：

```typescript
interface SectionPhase {
  agent: string          // Agent 名称，如 "架构师"、"安全审查"
  phase: string          // 阶段名称，如 "需求分析"、"API 设计"
  status: 'PENDING' | 'RUNNING' | 'COMPLETED' | 'FAILED' | 'SKIPPED'
  content?: string       // 阶段的产出文本
  thinking?: string      // 推理过程
  durationMs?: number    // 阶段耗时
  tokens?: number        // 消耗 token
  mode?: string          // 编排模式
  toolSummary?: Array<   // 工具调用汇总
    { name: string; count: number; status: string }
  >
  children?: SectionPhase[]  // 子阶段（递归）
}
```

### 3.2 最终回复文本的内联标注

最终汇总文本中，对关键产出添加轻量归属标记：

```
"我们采用了 RESTful 风格 (§架构师)，做了输入校验 (§安全审查)"
```

前端渲染规则：

| 标记 | 渲染效果 | 交互 |
|------|---------|------|
| `(§架构师)` | `🔍 架构师` badge | hover 浮层显示架构师阶段摘要 |
| `[N 个问题 → 查看]` | `⚠️ N 个问题` 链接 | 点击定位到对应阶段面板 |
| 普通文本 | 正常渲染 | — |

## 4. 后端改动

### 4.1 ForestConversationService 新增方法

```java
/**
 * 在 SSE 流中按阶段持久化产出到 message.sections。
 * 每个 TaskNode 完成后调用此方法追加阶段记录。
 */
void appendSection(UUID messageId, String agentName, String phaseName,
                   NodeStatus status, String content, String thinking,
                   long durationMs, int tokens) {
    Message message = messageRepository.findById(messageId);
    List<Map<String, Object>> sections = parseSections(message.getSections());

    Map<String, Object> section = new HashMap<>();
    section.put("agent", agentName);
    section.put("phase", phaseName);
    section.put("status", status.name());
    section.put("content", content);
    section.put("durationMs", durationMs);
    // ... 追加到列表

    message.setSections(toJson(sections));
    messageRepository.save(message);
}
```

### 4.2 SSE 事件增强

现有 `ForestEvent.detail(nodeId, forestId, accountId, message)` 不变。

在 `ForestConversationService.handleStreamingMessage()` 中，接到 `phase_end` 事件时调用 `appendSection()`，将当前阶段的 Agent 产出写入消息的 sections JSONB。

### 4.3 synthesis 最终消息

SynthesisNode（汇总节点）完成后，其输出成为 `message.content`（最终回复）。之前各阶段的产出通过 sections 携带。如果最终回复中需要引用阶段产出，由 synthesis Agent 自然产出 `(§Agent名)` 标记，后端不作后处理。

## 5. 前端改动

### 5.1 ChatBubble 增强

现有 `sections` 渲染已通过 `PhaseCard` 实现，需补充：

1. **Layer 1 时间线条** — 消息内容底部，缩略显示所有阶段及状态
2. **内联标注渲染** — `(§架构师)` → 可点击 badge
3. **默认折叠** — sections 面板默认收起，仅显示时间线条

### 5.2 PhaseCard 交互优化

| 现有 | 增强 |
|------|------|
| 始终展开 | 默认折叠（RUNNING 时展开） |
| 无层级指示 | 添加缩进线 |
| 无时间线条入口 | 点击时间线条节点跳转到对应 PhaseCard |

### 5.3 新增：InlineAgentBadge.vue

```vue
<template>
  <span class="agent-badge" @click.stop="showDetail = !showDetail"
        :title="`${agent} 的产出`">
    🔍 {{ agent }}
    <div v-if="showDetail" class="agent-badge__popover">
      {{ summary }}
    </div>
  </span>
</template>
```

## 6. 交互流程

```
用户发送消息 → SSE 开始推送
  │
  ├─ phase_start → 前端在 ChatBubble 中创建 sections 占位
  ├─ detail/thinking → 实时追加到当前阶段
  ├─ phase_end → 前端标记阶段完成，折叠
  │
  ├─ 最终汇总 → 写入 message.content
  │
  └─ 执行完成 → 显示完整三层：
      Layer 1: 最终回复文本（含 §标注）
      Layer 2: 折叠的编排详情（可展开）
      Layer 3: "查看执行详情" 链接 → ForestTree
```

## 9. 设计检查清单

- [ ] sections 结构是否向后兼容？→ 是，新增字段可选
- [ ] 对话面是否保持简洁？→ 是，Layer 1 默认，Layer 2 折叠
- [ ] 用户能否追溯到具体 Agent 的产出？→ 是，§badge + PhaseCard
- [ ] 是否与现有 ForestTree 共存？→ 是，Layer 3 入口保留
- [ ] SSE 协议是否无需改动？→ 是，仅消费侧增强
- [ ] 实时执行时用户能否看到进度？→ 是，RUNNING 阶段自动展开
- [ ] 实现是否需要 DB 迁移？→ 否，复用已有 JSONB
