## 1. 设计目标

### 1.1 解决的问题

森林架构的核心是一棵**目标执行树**，这棵树需要在以下五个维度同时满足需求：

| 维度 | 要求 | 典型操作 |
|---|---|---|
| **持久化** | 完整保存到 PostgreSQL，支持断点恢复 | 加载整棵树、更新单节点状态、批量写入 |
| **运行时遍历** | 执行引擎 DFS 遍历，高效率 | 父→子前序、子→父状态传播、兄弟遍历 |
| **动态修改** | 执行中可插入/替换子树 | HIERARCHICAL 模式规划阶段注入子任务 |
| **多态节点** | 7+ 种节点类型，各有不同数据和行为 | TaskNode、InnerGoalNode、MemoryNode... |
| **选择性压缩** | 不同类型节点不同折叠策略 | FoldStrategy 按 nodeType + 标签决策 |

### 1.2 核心设计决策

> **以邻接表为持久化模型，以对象引用图为运行时模型——两层同构，职责分离。**

```
持久化层 (PostgreSQL)                   运行时层 (Java Heap)
─────────────────────                  ───────────────────
                                       
forest_nodes 表                         TreeNode 对象图
  node_id        VARCHAR ←────映射────→ TreeNode.nodeId()
  parent_node_id VARCHAR ←────映射────→ TreeNode.parent() / children()
  sort_order     INT     ←────映射────→ TreeNode.order()
  node_type      VARCHAR ←────映射────→ TreeNode.nodeType() + accept(Visitor)
  status         VARCHAR ←────映射────→ ExecutableNode.status()
  mode           VARCHAR ←────映射────→ InnerGoalNode.mode()
  content        TEXT    ←────映射────→ ContentNode.content()
  metadata       JSONB   ←────映射────→ ContentNode.metadata()
  importance     DOUBLE  ←────映射────→ CompressibleNode.importance()

  ─── 写路径 ───                          ─── 执行路径 ───
  saveTree(root)  → 批量 INSERT          executeNode(node)
  updateNodeStatus → 单行 UPDATE         → children() 遍历
                                        → parent() 状态传播
```

## 2. 持久化层：邻接表模型

### 2.1 表结构

```sql
CREATE TABLE IF NOT EXISTS "forest_nodes" (
    -- ===== 树结构核心 =====
    "node_id"        VARCHAR     NOT NULL PRIMARY KEY,     -- 节点唯一 ID (UUID)
    "forest_id"      UUID        NOT NULL,                 -- 所属森林（聚合根）
    "parent_node_id" VARCHAR,                               -- 父节点 ID, NULL = 根
    "sort_order"     INT         NOT NULL DEFAULT 0,       -- 兄弟间顺序

    -- ===== 类型分派 =====
    "node_type"      VARCHAR(16) NOT NULL,                  -- task / inner_goal / synthesis / memory / ...

    -- ===== 可执行节点 (ExecutableNode) =====
    "mode"           VARCHAR(16),                           -- NONE / SEQUENTIAL / PARALLEL / CONDITIONAL / HIERARCHICAL / CONSENSUS
    "status"         VARCHAR(16),                           -- PENDING / RUNNING / COMPLETED / FAILED / CANCELLED

    -- ===== 内容节点 (ContentNode) =====
    "content"        TEXT,                                  -- 节点文本内容
    "metadata"       JSONB       DEFAULT '{}'::jsonb,       -- 元数据 KV

    -- ===== 可压缩节点 (CompressibleNode) =====
    "importance"     DOUBLE PRECISION,                      -- 重要性 [0, 1]
    "estimate_tokens" BIGINT,                               -- 子树 token 估算

    -- ===== 审计 =====
    "kind"           VARCHAR(16) DEFAULT 'INSTANCE',        -- INSTANCE / TEMPLATE
    "content_hash"   VARCHAR(64),                           -- 内容哈希（增量更新判断）
    "completed_at"   TIMESTAMPTZ(6),
    "updated_at"     TIMESTAMPTZ(6) NOT NULL DEFAULT now(),

    -- ===== 约束 =====
    CONSTRAINT "forest_nodes_forest_id_fkey"
        FOREIGN KEY ("forest_id") REFERENCES "forests"("forest_id") ON DELETE CASCADE,
    CONSTRAINT "forest_nodes_parent_node_id_fkey"
        FOREIGN KEY ("parent_node_id") REFERENCES "forest_nodes"("node_id") ON DELETE CASCADE
);
```

### 2.3 为什么用邻接表（而不是其他树存储模型）

| 模型 | 子树查询 | 祖先查询 | 插入/删除 | 适合我们的理由 |
|---|---|---|---|---|
| **邻接表（当前）** | 递归 CTE O(n) | 递归 CTE O(n) | O(1) | 树深度 ≤ 100，递归 CTE 足够快；动态插入零成本，最适合"执行中变更"场景 |
| **嵌套集 (Nested Set)** | O(1) 范围查询 | O(1) | O(n) 重编号 | **不适用**：每次插入/移动节点需要重编号大量行，与 HIERARCHICAL 模式动态规划冲突 |
| **物化路径 (Materialized Path)** | 前缀匹配 O(log n) | 路径分割 O(1) | O(d) 更新后代路径 | **可选的辅助加速**：作为冗余列加速祖先查询，主结构仍用邻接表 |
| **闭包表 (Closure Table)** | 按 ancestor 查询 O(log n) | 按 descendant 查询 O(log n) | O(n²) | **不适用**：插入一个节点需要插入 d 行闭包记录，频繁插入时不可接受 |

**关键决策依据**：我们每执行一次 `findSubtree`（完整树加载）做一次递归 CTE；但在执行过程中，子节点变化是常态。邻接表是唯一在**动态插入上零成本**的模型。

### 2.4 树加载：递归 CTE

```sql
-- 一次 CTE 加载整棵子树（当前实现）
WITH RECURSIVE subtree AS (
    -- 锚点：根节点
    SELECT node_id, parent_node_id, node_type, sort_order, mode, status,
           content, metadata, importance, estimate_tokens, kind, content_hash,
           completed_at, updated_at,
           0 AS depth
    FROM forest_nodes
    WHERE node_id = :rootNodeId AND forest_id = :forestId

    UNION ALL

    -- 递归：子节点
    SELECT n.node_id, n.parent_node_id, n.node_type, n.sort_order, n.mode,
           n.status, n.content, n.metadata, n.importance, n.estimate_tokens,
           n.kind, n.content_hash, n.completed_at, n.updated_at,
           s.depth + 1
    FROM forest_nodes n
    INNER JOIN subtree s ON n.parent_node_id = s.node_id
    WHERE n.forest_id = :forestId
)
SELECT * FROM subtree ORDER BY depth, sort_order;
```

**为什么不用 N+1 查询逐层取**：树的深度 ≤ 100，CTE 在 PostgreSQL 16 中经过优化，单次往返比 N+1 快一个数量级。`assembleTree()` 在内存中以 O(n) 两遍扫描构建对象引用图。

## 3. 运行时层：对象引用图

### 3.1 核心接口层次

```java
// 第0层：树节点基接口
interface TreeNode {
    String nodeId();
    TreeNode parent();
    void setParent(TreeNode parent);
    List<TreeNode> children();
    int order();
    void setOrder(int order);
    default boolean isLeaf() { return children().isEmpty(); }
    default boolean isExecutable() { return false; }
    default String nodeType() { ... }
    void accept(ForestVisitor visitor);   // 编译期安全的类型分派
}

// 第1层：三个正交的能力接口
interface ExecutableNode extends TreeNode {
    Mode mode();                          // 编排模式
    NodeStatus status();                  // 执行状态
    void setStatus(NodeStatus status);
}

interface ContentNode extends TreeNode {
    String content();                     // 文本内容
    Map<String, Object> metadata();       // 元数据 KV
    default Set<String> tags() { return Set.of(); }  // 标签（用于压缩策略识别）
}

interface CompressibleNode extends TreeNode {
    double importance();                  // 重要性 [0, 1]
    long estimateSelfTokens();            // 自身 token 估算
    long estimateSubtreeTokens();         // 子树 token 估算
    void invalidateTokenCache();          // 缓存失效
    default boolean isFoldable() { return importance() < 0.3; }

    // 状态变更 → 父链缓存失效（O(log n)）
    default void onStatusChanged() {
        invalidateTokenCache();
        TreeNode p = parent();
        while (p != null) {
            if (p instanceof CompressibleNode cn) cn.invalidateTokenCache();
            p = p.parent();
        }
    }
}

// 第2层：具体节点类型（8种）
class InnerGoalNode  implements ExecutableNode, CompressibleNode { ... }
class TaskNode       implements ExecutableNode, CompressibleNode, ContentNode { ... }
class SynthesisNode  implements ExecutableNode, ContentNode { ... }
class MessageNode    implements ContentNode { ... }
class MemoryNode     implements CompressibleNode, ContentNode { ... }
class FileSnapshotNode implements ContentNode { ... }
class ContextBlockNode  implements ContentNode { ... }
class SearchKBNode   implements ExecutableNode, ContentNode { ... }
```

### 3.2 对象引用图的构建过程

```java
// 步骤1：扁平行列表 → Map（assembleTree 的第一遍扫描）
Map<String, TreeNode> nodeMap = new LinkedHashMap<>();
Map<String, List<String>> parentToChildren = new HashMap<>();

for (ForestNodeEntity row : rows) {
    TreeNode node = createNode(row);           // 按 nodeType 构造具体子类
    nodeMap.put(row.getNodeId(), node);
    if (row.getParentNodeId() != null) {
        parentToChildren.computeIfAbsent(
            row.getParentNodeId(), k -> new ArrayList<>()
        ).add(row.getNodeId());
    }
}

// 步骤2：连接父子引用（第二遍扫描）
for (Map.Entry<String, List<String>> entry : parentToChildren.entrySet()) {
    TreeNode parent = nodeMap.get(entry.getKey());
    if (parent instanceof InnerGoalNode ig) {
        List<TreeNode> children = entry.getValue().stream()
            .map(nodeMap::get)
            .toList();
        // 设置父引用 + 兄弟序号
        for (int i = 0; i < children.size(); i++) {
            children.get(i).setParent(parent);
            children.get(i).setOrder(i);
        }
        ig.replaceChildren(children);          // 注入不可变子节点列表
    }
}
```

```
                    ┌───────────────────┐
                    │  InnerGoalNode    │  ← 根节点，无 parent
                    │  mode=SEQUENTIAL  │
                    │  status=RUNNING   │
                    │  children=[*,*]   │
                    └───┬───────┬───────┘
                        │       │
          ┌─────────────┘       └─────────────┐
          ▼                                    ▼
┌──────────────────────┐          ┌──────────────────────┐
│  InnerGoalNode       │          │  InnerGoalNode       │
│  mode=PARALLEL       │          │  mode=CONSENSUS      │
│  status=COMPLETED    │          │  status=PENDING      │
│  parent=↑            │          │  parent=↑            │
│  children=[*,*,*]    │          │  children=[*]        │
│  importance=0.7      │          │  importance=0.5      │
└──┬───────┬───────┬───┘          └───┬──────────────────┘
   │       │       │                  │
   ▼       ▼       ▼                  ▼
 ┌────┐ ┌────┐ ┌────┐          ┌──────────┐
 │Task│ │Task│ │Task│          │Synthesis │

```

### 3.3 对象引用 vs 扁平数组的取舍

| 方案 | 优点 | 缺点 | 当前选择理由 |
|---|---|---|---|
| **对象引用图** | 面向对象自然；节点携带多态行为；`parent()`/`children()` 直接跳转；便于 `Visitor` 分派 | 缓存不连续（堆上分散对象）；每个对象有 Header 开销 (~16-24 bytes) | ✅ 核心选择。树的规模小（数十到数百节点），对象开销不计；代码可读性和扩展性更重要 |
| **扁平数组+索引** | 缓存友好（连续内存）；遍历性能更高（数组下标）；序列化简单 | 失去多态（所有节点在一个数组中）；插入需要移位；父引用需独立数组 | ⚠️ 备选。当单棵树节点数 > 10000 且 DFS 成为热点时考虑。当前不需要 |

## 4. 操作映射：邻接表 + 对象引用如何覆盖所有访问模式

### 4.1 核心操作映射矩阵

```java
// ───────── 读操作 ─────────

// 操作1：加载整棵树
// 数据库：递归 CTE（单次 SQL 往返）
// 运行时：assembleTree() 两遍扫描 O(n)
TreeNode root = assembleTree(cteResult);

// 操作2：前序遍历（Depth-First Search）← 最核心操作
// 运行时：递归调用 children()
void executeNode(ExecutableNode node, ...) {
    for (TreeNode child : node.children()) {    // O(#children)
        if (child instanceof ExecutableNode e) {
            executeNode(e, ...);                // DFS 递归
        }
    }
}
// 复杂度：O(N) 其中 N 为节点总数

// 操作3：子→父传播
// 运行时：parent() 指针跳转
void onStatusChanged() {
    TreeNode p = parent();                       // O(1) 指针跳转
    while (p != null) {
        if (p instanceof CompressibleNode cn) cn.invalidateTokenCache();
        p = p.parent();
    }
}
// 复杂度：O(depth)，最坏 O(100)（深度限制）

// 操作4：按 ID 查找节点
// 运行时：不使用。执行引擎通过引用访问节点。
// 数据库：SELECT * FROM forest_nodes WHERE node_id = :id
// 复杂度：O(1) 主键索引

// 操作5：查找下一个待执行兄弟节点
// 运行时：遍历 parent().children()，找到 order 大于当前的最小的 PENDING 节点
// 数据库：SELECT * FROM forest_nodes
//         WHERE parent_node_id = :pid AND forest_id = :fid
//           AND sort_order > :currentOrder AND status = 'PENDING'
//         ORDER BY sort_order LIMIT 1
// 复杂度：O(#siblings) 内存 / O(log n) 复合索引


// ───────── 写操作 ─────────

// 操作6：插入子节点
// 运行时：replaceChildren() / addChild()
// 数据库：INSERT INTO forest_nodes (...) VALUES (...)
// 复杂度：O(1) 新建 + O(1) 插入列表

// 操作7：更新节点状态
// 运行时：setStatus(newStatus) → onStatusChanged()
// 数据库：UPDATE forest_nodes SET status = :s, updated_at = now() WHERE node_id = :id
// 复杂度：O(1) 内存 + O(1) 数据库单行更新

// 操作8：批量写入整棵树
// 运行时：collectNodes() DFS 收集扁平列表
// 数据库：先 DELETE 旧子树，再 saveAll(entities) 批量 INSERT
// 复杂度：O(N)

// 操作9：删除子树（折叠/失败回滚）
// 运行时：parent().children().remove(node)
// 数据库：DELETE FROM forest_nodes WHERE path LIKE 'root_id/%' （有 Materialized Path 时）
//        或递归 CTE 删除
// 复杂度：O(#subtree_nodes)
```

### 4.2 五种编排模式对树结构的遍历方式

每种编排模式本质上是不同的子节点遍历策略。对象引用图中的 `children()` 列表是所有这些策略的输入：

```java
// ─── SEQUENTIAL（顺序串行）───
// 遍历方式：按 order 顺序逐个 execute
for (int i = 0; i < node.children().size(); i++) {
    ExecutableNode child = (ExecutableNode) node.children().get(i);
    yield* executeNode(child, ctx, depth + 1); // 同步等待前一个完成
}

// ─── PARALLEL（并行独立）───
// 遍历方式：同时 execute 所有子节点
Flux.fromIterable(node.children())
    .flatMap(child -> executeNode((ExecutableNode) child, ctx, depth + 1))
    // flatMap 并发执行

// ─── CONDITIONAL（条件路由）───
// 遍历方式：按阶段执行，每阶段后 LLM 决策是否继续
int stage = 0;
while (stage < node.children().size()) {
    ExecutableNode child = (ExecutableNode) node.children().get(stage);
    yield* executeNode(child, ctx, depth + 1);
    boolean shouldContinue = llm.decide(...);    // LLM 决策
    if (!shouldContinue) break;
    stage++;
}

// ─── HIERARCHICAL（层级分解）───
// 遍历方式：第一步是规划（生成子节点拓扑），后续按 DAG 拓扑序执行
// 关键：规划阶段会修改 children() 列表——动态注入新节点
ExecutableNode planner = (ExecutableNode) node.children().get(0);
yield* executeNode(planner, ctx, depth + 1);
// 规划器修改了 node.children()，添加了新的子节点

List<ExecutableNode> dagOrder = topologicalSort(node.children());
for (ExecutableNode child : dagOrder) {
    yield* executeNode(child, ctx, depth + 1);
}

// ─── CONSENSUS（共识决策）───
// 遍历方式：先并行执行所有常规子节点，再执行合成节点
List<ExecutableNode> regular = regularChildren(node.children());
List<ExecutableNode> synthesis = synthesisChildren(node.children());

yield* parallel(regular.map(c -> executeNode(c, ...)));
// 汇聚 + 调用 LLM 合成
yield* synthesis.map(s -> executeNode(s, ...));
```

## 5. 选择性压缩：压缩策略与树结构的交互

### 5.1 压缩流程

```java
// ForestCompressor 的核心控制流
void compressNode(TreeNode node, TokenBudget budget, int depth) {
    // 先深度优先递归子节点
    for (TreeNode child : node.children()) {
        compressNode(child, budget, depth + 1);
    }

    // 只处理 CompressibleNode
    if (!(node instanceof CompressibleNode cn)) return;

    // 按 nodeType 查找策略
    FoldStrategy strategy = strategies.get(node.nodeType());
    if (strategy == null) return;

    // 执行策略决策
    FoldDecision decision = strategy.decide(node, budget);
    if (decision.shouldFold()) {
        cn.invalidateTokenCache();         // 标记缓存失效
        // 具体"折叠"行为由调用方定义：
        //   1. 替换 content 为摘要（保持节点存在）
        //   2. 标记为 HIDDEN（前端隐藏）
        //   3. 裁剪子节点（删除可丢弃的孙节点）
    }
}
```

### 5.2 节点粒度拆解：让工具调用不被压缩

当前 `TaskNode` 将 ReAct 循环的所有内容拼接在一个 `content` 中：

```java
// 当前：粗粒度
TaskNode {
    content: "Thought: 需要搜索X\nAction: search\nActionInput: X\nObservation: 结果A\nThought: 需要读取\nAction: read\nObservation: 结果B\nThought: 综合得出最终答案Y\nFinal: Y"
}
// → 压缩时只能"整节点保留或整节点折叠"

// 优化：细粒度
InnerGoalNode (mode=SEQUENTIAL, tags=["REACT_CYCLE"])
  ├── ToolCallNode      { content: "search X",    tags: ["TOOL_CALL"] }
  ├── ToolResultNode    { content: "结果A",        tags: ["TOOL_RESULT"] }
  ├── ToolCallNode      { content: "read URL",    tags: ["TOOL_CALL"] }
  ├── ToolResultNode    { content: "结果B",        tags: ["TOOL_RESULT"] }
  └── SynthesisNode     { content: "最终答案: Y",  tags: ["SUMMARY"] }
      ↑ 只有 SynthesisNode 被折叠，ToolCall/Result 保留
```

对应的 `FoldStrategy`：

```java
@Component
public class ToolCallFoldStrategy implements FoldStrategy {
    @Override public String supportedType() { return "tool_call"; }
    @Override public FoldDecision decide(TreeNode node, TokenBudget budget) {
        return FoldDecision.skip("工具调用记录，必须保留");
    }
}

@Component
public class ToolResultFoldStrategy implements FoldStrategy {
    @Override public String supportedType() { return "tool_result"; }
    @Override public FoldDecision decide(TreeNode node, TokenBudget budget) {
        return FoldDecision.skip("工具执行结果，必须保留");
    }
}

@Component
public class ReactionSynthesisFoldStrategy implements FoldStrategy {
    @Override public String supportedType() { return "synthesis"; }
    @Override public FoldDecision decide(TreeNode node, TokenBudget budget) {
        // 已完成且重要性低 → 折叠为摘要
        if (node.status() == COMPLETED && node.importance() < 0.5) {
            return FoldDecision.fold(truncate(node.content(), 50));
        }
        return FoldDecision.skip("保留");
    }
}
```

### 5.3 标签系统：灵活的类型无关过滤

除了按 `nodeType` 匹配策略外，增加标签维度：

```java
// 在 ContentNode 或 TreeNode 上增加 tags
interface TreeNode {
    // ...
    default Set<String> tags() { return Set.of(); }
}

// 使用场景
Set<String> KEEP_TAGS = Set.of("TOOL_CALL", "TOOL_RESULT", "ERROR", "HITL_DECISION");

// FoldStrategy 的全局守卫
if (node.tags().stream().anyMatch(KEEP_TAGS::contains)) {
    return FoldDecision.skip("标签标记为保留");
}
```

## 6. 优化建议（基于当前邻接表 + 对象引用的增强）

### 6.2 P1：增量 `addChild` + 内存优先的 `findNextSibling`

**目的**：解决场景一（动态分解时兄弟节点在内存中而非 DB）

```java
// InnerGoalNode 增加增量添加
public class InnerGoalNode {
    private List<TreeNode> children;  // 当前是 unmodifiable，改为可变的

    public void addChild(TreeNode child) {
        child.setParent(this);
        child.setOrder(children.size());
        // 创建可变的 children 列表（如果当前是 unmodifiable）
        if (children.getClass().getName().contains("Unmodifiable")) {
            children = new ArrayList<>(children);
        }
        children.add(child);
    }
}

// ForestRepositoryAdapter.findNextSibling 增加内存路径
@Override
public TreeNode findNextSibling(String nodeId, UUID forestId, UUID accountId) {
    Optional<TreeNode> memoryResult = findNextSiblingInMemory(nodeId);
    if (memoryResult.isPresent()) return memoryResult.get();
    // 回退到 DB 查询
    return findNextSiblingFromDB(nodeId, forestId, accountId);
}
```

### 6.2 P2：可选的 Materialized Path 列

**目的**：加速祖先查询（当前递归 CTE 在深度 ≤ 100 时已足够，此优化长期储备）

```sql
-- 使用 PostgreSQL ltree 扩展
CREATE EXTENSION IF NOT EXISTS ltree;

-- 新增路径列
ALTER TABLE forest_nodes ADD COLUMN path_ltree LTREE;

-- 插入/移动节点时维护 path（触发器或应用层）
-- path = parent.path || node_id

-- 查询所有祖先（不再需要反向递归 CTE）
SELECT * FROM forest_nodes
WHERE path_ltree @> (SELECT path_ltree FROM forest_nodes WHERE node_id = :target);
-- @> 操作符：左参数是右参数的祖先

-- 查询所有子孙（不再需要递归 CTE）
SELECT * FROM forest_nodes
WHERE path_ltree <@ (SELECT path_ltree FROM forest_nodes WHERE node_id = :root);
-- <@ 操作符：左参数是右参数的后代

-- GIST 索引加速
CREATE INDEX "idx_forest_nodes_path_gist" ON forest_nodes USING GIST ("path_ltree");
```

## 7. 数据一致性保证

### 7.1 两层一致性：内存 + 数据库

```java
// 执行过程中的状态变更策略：
// 1. 先写内存（立即生效，执行引擎可见）
// 2. 后写数据库（异步持久化，断点恢复用）

// 为什么不是事务性的两阶段提交？
// - 执行引擎在内存中运行，不依赖 DB 事务完成状态变更
// - 内存中的状态变更失败 = 整个执行失败（由 Reactor error 处理）
// - DB 只做"兜底持久化"——即使系统崩溃，已写盘的状态可恢复
// - 这本质上是 "Last Write Wins" 的最终一致性模型

void setStatus(NodeStatus newStatus) {
    this.status = newStatus;                           // 立即生效
    onStatusChanged();                                  // O(log n) 缓存失效
    forestRepository.updateNodeStatus(nodeId, newStatus); // 异步写 DB
    // 注意：这里不等待 DB 返回，通过 Reactor 的事件循环调度
}
```

### 7.2 断点恢复流程

```java
// 1. 系统重启后扫描所有 status = RUNNING 的根节点
List<TreeNode> roots = forestRepository.findRootNodesByStatus(RUNNING, accountId);

// 2. 对每个根节点：
//    a. findSubtree(rootNodeId) 加载完整树
//    b. 从根开始遍历，找到第一个 status != COMPLETED 的节点（断点）
//    c. 从这个节点恢复执行
//    d. 使用 findNextSibling 或者 ModeStrategy 的位置恢复逻辑

// 3. 执行状态信任链：
//    DB 中的 status 是权威来源
//    内存中 status 以 DB 加载为准
//    执行过程中 status 以 setStatus 更新为准
```

## 8. 与其他候选方案的对比总结

```java
// ───────── 邻接表 + 对象引用（选中方案）─────────
// 优点：
//   1. 动态插入零成本——执行中可任意修改树结构
//   2. 父→子和子→父都是 O(1) 指针操作
//   3. type 分派通过 Visitor 编译期安全
//   4. 两遍扫描加载整棵树，后续所有操作在内存中
// 缺点：
//   1. 祖先/后代查询需要递归 CTE（深度 ≤ 100 时不是问题）
//   2. 对象在堆上分散，缓存不友好（树规模小时不计）
// 适用：所有需要动态修改 + 运行时遍历的场景


// ───────── 压缩基数树（Patricia Trie）─────────
// 优点：前缀匹配极快
// 缺点：
//   1. 无 parent 指针——父节点传播需遍历整棵树
//   2. 无 children 列表——兄弟遍历需前缀查询
//   3. 无 order 概念——排序依赖 key 的字典序
//   4. 节点分裂复杂——动态插入有开销
//   5. 无法表达节点多态——key 不携带领域类型
// 适用：IP 路由、自动补全、字典——不是任务编排


// ───────── B+树─────────
// 优点：
//   1. 磁盘 I/O 优化——大扇出适合外存
//   2. 范围查询——排序 key 的区间检索
// 缺点：
//   1. 层次编码到 key——插入中间层需重编码所有后代
//   2. 无面向对象语义——无法表达 parent/children/type 关系
//   3. 动态插入可能触发页分裂——执行中性能不稳定
//   4. PostgreSQL 索引层已经是 B+树——再加一层纯属冗余
// 适用：数据库索引、文件系统——不是领域树结构


// ───────── 最终评价─────────
// 场景              邻接表+对象引用  压缩基数树  B+树
// 动态分解(P0)      ✅ 天然          ❌ 分裂复杂  ❌ key 重编码
// 选择压缩(P0)      ✅ 策略可扩展     ❌ 无折叠    ❌ 无折叠
// 记忆聚合(P1)      ✅ 跨树引用可加   ❌ 无摘要    ❌ 不解决
// 持久化             ✅ 标准 SQL      ❌ 序列化难  ⚠️ 但冗余
// 运行时性能         ✅ 内存引用       ❌ 前缀搜索  ❌ 查 key
// 多态节点           ✅ Visitor       ❌          ❌

// 结论：邻接表 + 对象引用是正确的选择。
// 优化方向不应该是替换它，而是以它为基座增加标签系统、增量 addChild、跨树引用。
```

## 9. 附录

### 9.1 术语表

| 术语 | 定义 |
|---|---|
| **邻接表 (Adjacency List)** | 每行记录 `(node_id, parent_node_id)` 的树存储模型。每个节点只记录它的直接父节点 |
| **对象引用图 (Object Reference Graph)** | 运行时内存中，节点通过 `parent()` 和 `children()` 直接相互引用形成的对象图 |
| **Materialized Path** | 冗余存储从根到当前节点的完整路径字符串，用于加速祖先/后代查询 |
| **递归 CTE** | PostgreSQL 的 `WITH RECURSIVE` 语法，用于在 SQL 中遍历树结构 |
| **`assembleTree`** | 将扁平的行列表构建为对象引用图的两遍扫描算法 |
| **`collectNodes`** | 将对象引用图 DFS 扁平化为行列表（`saveTree` 的逆操作） |
