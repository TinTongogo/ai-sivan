# 第十二章：可观测性

## 核心问题

多 Agent 协作系统是一个典型的复杂分布式系统。当某个 Agent 执行失败、或整个编排挂起时，如何快速定位问题？

## 12.1 执行链路追踪

### 12.1.1 Span/Trace 模型

每次用户请求 = 一个 Trace，每个节点执行 = 一个 Span：

```
Trace: conversationId
├── Span: TaskNode "分析项目"（根节点）
│   ├── Span: InnerGoalNode PARALLEL
│   │   ├── Span: TaskNode "分析技术"（Agent A）
│   │   │   ├── Span: ToolCall "file_read"（读文件）
│   │   │   └── Span: ToolCall "bash"（执行命令）
│   │   └── Span: TaskNode "分析市场"（Agent B）
│   └── Span: SynthesisNode（汇总）
```

```java
public record Span(
    String traceId,
    String parentSpanId,
    String spanId,
    String nodeId,
    String nodeType,
    Instant startTime,
    Instant endTime,
    String status,
    ...
) {}
```

### 12.1.2 导出

Span 通过 `SpanExporter` 接口导出：
- 当前实现：日志文件（JSON Lines 格式）
- 可扩展：Jaeger、Zipkin 等

### 12.1.3 采样决策

采样率按节点类型动态调整：

| 节点模式 | 采样率 |
|---------|--------|
| SUMMARY（根节点） | 100% |
| CONDITIONAL | 100% |
| CONSENSUS | 100% |
| SEQUENTIAL | 50% |
| HIERARCHICAL | 50% |
| PARALLEL | 10% |

## 12.2 指标采集

### 12.2.1 ForestMetricsCollector

`ForestMetricsCollector` 采集执行粒度的指标：

- `totalTokens`：每次执行的 Token 消耗
- `durationMs`：每次执行的耗时
- `modelName`：使用的模型
- `nodeType`：节点类型
- `success/fail`：执行结果

### 12.2.2 系统级指标

| 指标 | 采集点 | 用途 |
|------|--------|------|
| 每分钟请求数 | StreamManager | 负载监控 |
| 平均响应时间 | MetricsCollector | 性能监控 |
| Token 消耗率 | TokenUsageService | 成本监控 |
| 路由各 Tier 命中率 | RoutingDecisionRecorder | 路由效果评估 |
| 工具调用成功率 | IToolUsageRepository | 工具健康度 |
| 记忆召回率 | MemoryRepositoryAdapter | 记忆系统效果 |

### 12.2.3 输出方式

- 日志：结构化日志（JSON 格式），按 `conversationId` 关联
- 内存：`MetricsCollector` 维护最近 1 小时的聚合数据
- API：通过 `/v2/metrics` 端点暴露 Prometheus 格式

## 12.3 SSE 进度推送

### 12.3.1 事件类型

```java
enum EventType {
    DETAIL,       // 文本内容块
    THINKING,     // 思考过程
    TOOL_CALL,    // 工具调用
    TOOL_RESULT,  // 工具结果
    MILESTONE,    // 执行里程碑
    PHASE_BEGIN,  // 编排阶段开始
    PHASE_END,    // 编排阶段结束
    ERROR         // 错误
}
```

### 12.3.2 进度推算

`ProgressAggregator` 根据执行树的叶子节点数量推算总进度：

```java
// 推导总叶子数（递归）
int totalLeaves = countLeaves(tree);
// 推导已完成的叶子数
int completedLeaves = countCompletedLeaves(tree);
// 进度百分比
int progress = completedLeaves × 100 / totalLeaves;
```

前端 PipelineDialog 展示：
```
[2/5] 60% —— 表示 5 个叶子节点中 2 个已完成
```

## 12.4 execution_logs 表

所有节点状态变更写入 `forest_execution_logs`：

| 列 | 说明 |
|----|------|
| log_id | 主键 |
| forest_id | 森林 ID |
| node_id | 节点 ID |
| event_type | LIFECYCLE / TOOL_CALL / ERROR |
| status_before | 变更前状态 |
| status_after | 变更后状态 |
| message | 事件描述 |
| metadata | JSONB 扩展信息 |
| created_at | 创建时间 |

**审计功能**：通过 execution_logs 可以完整重建一次执行的全部状态变化。

## 12.5 HITL（Human-in-the-Loop）

当节点 `status='PAUSED'` 时（HITL 审批节点），流程暂停等待人工干预：

```java
// HITL 审批记录在 forest_hitl_reviews 表中
// 前端 PipelineDialog 显示暂停节点
// 用户选择：批准（继续）或拒绝（取消）

// approveHitl(forestId, nodeId)
//   → 更新节点状态为 PENDING
//     → RecoveryManager 检测到节点可执行
//       → 继续执行

// rejectHitl(forestId, nodeId)
//   → 更新节点状态为 CANCELLED
//     → 跳过该节点，继续执行后续节点
```
