# 第九章：上下文压缩

## 核心问题

LLM 的上下文窗口是有限的（Qwen3 支持 128K tokens）。当对话历史超过窗口大小时，如何在不丢失关键信息的前提下压缩上下文？

## 9.1 三层压缩管道

### 9.1.1 HOT 层（热数据）
- 最近 N 轮（默认 3 轮）的原始消息
- 不压缩，完整保留
- 保证 LLM 看到最近的对话上下文

### 9.1.2 WARM 层（温数据）
- 超出 HOT 范围的历史消息
- 使用 LLM 自动摘要（`ExtractionService`）
- 摘要策略：提取关键信息、忽略无关细节
- 每次压缩后更新摘要

### 9.1.3 COLD 层（冷数据）
- 更早的历史消息
- 使用向量检索从记忆库召回
- 仅在需要时注入（通过 FlashbackScanner）
- COLD 层触发条件：Token 预算不足时

## 9.2 Token 预算分配

### 9.2.1 预算计算

```java
// PromptContextService 中计算可用 token 数
int maxPromptTokens = contextLength × budgetRatio;
// 例如：128K × 0.7 = 89.6K tokens 可用
```

### 9.2.2 按场景分配

| 场景 | 预算比例 | 说明 |
|------|---------|------|
| CHAT | 0.7 | 对话为主，预留 30% 给回复 |
| SQUAD | 0.6 | 多 Agent 协作，每个 Agent 需要更多空间 |
| KB | 0.5 | 知识库场景，需要大量 token 放检索结果 |
| TOOL | 0.5 | 工具调用场景，工具描述占用较多 |

### 9.2.3 压缩触发

`ConversationCompressionService` 在每次构建消息前判断：
```
当前消息总 token > 预算 → 触发 HOT 层压缩
HOT 压缩后仍超预算 → 触发 WARM 层摘要
WARM 后仍超预算 → 触发 COLD 层记忆归档
```

## 9.3 折叠策略（FoldStrategy）

### 9.3.1 接口

```java
public interface FoldStrategy {
    String supportedType();
    FoldDecision decide(TreeNode node, TokenBudget budget);
}
```

### 9.3.2 各节点类型的折叠策略

| 策略 | 应用节点 | 逻辑 |
|------|---------|------|
| `MessageFoldStrategy` | message | 低重要性消息 → 摘要；高重要性 → 保留 |
| `MemoryFoldStrategy` | memory | importance < 0.3 → 折叠；超预算 → 摘要 |
| `TaskFoldStrategy` | task | 已完成任务 → 保留结果摘要 |
| `InnerGoalFoldStrategy` | inner_goal | 子节点全部完成 → 折叠为结果摘要 |

### 9.3.3 折叠决策

```java
sealed interface FoldDecision permits FoldDecision.Skip, FoldDecision.Fold {
    record Skip(String reason) implements FoldDecision {}
    record Fold(String replacement, String summary) implements FoldDecision {}
}
```

- `Skip`：不折叠（重要性高或预算充足）
- `Fold`：折叠为摘要（用 replacement 替换原始内容）

### 9.3.4 预算传播

`TreeCompressor` 从根节点向下遍历：
```
根节点获得总预算
  → 按子节点数量和重要性分配预算
    → 每个子节点在自己的预算内决定是否折叠
      → 折叠后释放的 token 回流到父节点
```

## 9.4 CompressionService

### 9.4.1 压缩时机

- 对话保存后：`ConversationCompressionService.compress()`
- 执行树构建前：`ForestCompressor.compress()`
- 记忆写入前：`MemoryFoldStrategy.decide()`

### 9.4.2 ExtractionService

负责实际的 LLM 摘要调用：

```java
// 对一段文本生成摘要
String summary = extractionService.extract(
    content,                    // 原文
    maxTokens,                  // 摘要最大 token
    ExtractionService.Mode.SUMMARY  // 摘要模式
);
```

支持模式：
- `SUMMARY`：通用摘要
- `KEY_POINTS`：提取关键要点
- `TITLE`：生成标题
- `QUOTE`：提取引文

## 关键设计决策

- **为什么不用固定长度的滑动窗口？** 信息密度不均衡——有些消息虽然旧但重要（如用户指定的关键约束），滑动窗口可能丢弃它们
- **为什么折叠后允许重建？** 基于重要性和预算的折叠是可逆的——如果需要，可以从记忆库中召回原始内容
- **预算为什么按比例分配而不是均分？** 不同场景对上下文的需求不同，CHAT 场景需要更多对话历史，TOOL 场景需要更多工具描述
