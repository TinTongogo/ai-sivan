# 第一章：对话系统

## 核心问题

用户的每次输入如何被系统接收、处理、并流式返回？在单次对话中，系统如何支持多轮上下文、消息编辑、重新生成？

## 1.1 消息模型

### 1.1.1 Message 领域实体
- `Message` 类的核心字段：role（USER/ASSISTANT/SYSTEM）、content、thinking、images、attachments
- 消息的状态机：PENDING → COMPLETED / FAILED
- 消息的排序模型：sortOrder 游标分页

### 1.1.2 Msg（LLM 消息）与 Message（领域消息）的区别
- `Msg` 定义在 `sivan-core-api` 中，是 LLM 的消息格式（Role + Content 列表）
- `Message` 定义在 `sivan-domain/conversation` 中，是业务实体（含 metadata、附件、模型信息）
- 两者的转换时机：`PromptContextService` 在调用 LLM 前将 Message 转为 Msg

### 1.1.3 消息的持久化双写历史
- 旧方案：独立的 `messages` 表 + `forest_nodes` 表
- V25 迁移后：全部合入 `forest_nodes(node_type='message')`
- MessageRepositoryAdapter 的适配模式

## 1.2 流式输出

### 1.2.1 SSE 架构
- Spring WebFlux + SseEmitter
- 事件类型：response（内容块）、thinking（思考过程）、meta（元信息：模型名、token 数）、phase（编排阶段进度）
- `SseFormatter` 的 JSON 事件格式化

### 1.2.2 客户端接收
- 前端的 `EventSource` / 自定义 SSE 客户端
- `useChatStream` composable 的增量文本拼接
- 中断与恢复：客户端断开后后台继续执行

## 1.3 多轮上下文

### 1.3.1 上下文构建
- `PromptContextService.buildLlmMessages()` 的构建链路
- 历史消息的加载策略：游标分页（`findMessagesBeforeSortOrder`）
- 系统提示词的注入

### 1.3.2 上下文压缩介入
- `ConversationCompressionService` 在消息构建前的压缩触发
- 三层压缩策略：HOT（近 3 轮）→ WARM（摘要）→ COLD（记忆）

## 1.4 重排与编辑

### 1.4.1 重新生成（Regenerate）
- 找到目标消息的上一条用户消息作为输入
- 从该点重建执行树
- 覆盖旧消息的显示（不删除持久化数据）

### 1.4.2 消息编辑
- 编辑后重新生成执行链路
- 分支处理：是否保留后续消息？

## 1.5 与森林架构的交互

- **消息是执行树的叶子节点**：每条消息通过 `parent_node_id` 链接到其执行树根
- **但默认 `resolveParentNodeId` 将 parent 设为 conversationId**，导致消息平铺在对话容器下而非挂在执行树下
- `linkMessageToTree()` 的修复历程：save 前调用 → save 后调用

## 关键代码路径

```
用户输入 → MessageCrudService.save() → ForestConversationService.send()
  → PromptContextService.buildLlmMessages()（含压缩）
    → ForestExecutionOrchestrator.buildTree()
      → ForestExecutionOrchestrator.executeTree()
        → LeafExecutor.execute() → Flux<ForestEvent> → SSE sink
```
