# 第七章：叶子节点体系

## 核心问题

执行树的叶子节点（TaskNode）由 LeafExecutor 执行。不同类型的任务需要不同的执行策略——对话、图像生成、语音识别等。LeafExecutor 体系如何做到可扩展？

## 7.1 LeafExecutor 接口

### 7.1.1 接口定义

```java
public interface LeafExecutor {
    String supportedType();           // 支持哪种 node_type
    Flux<ForestEvent> execute(TreeNode node, ExecutionContext ctx, EventSink sink);
    default int maxRetries() { return 0; }
}
```

- `supportedType()` 返回节点类型标识（如 `"task"`、`"message"`、`"image_gen"`）
- `execute()` 返回 `Flux<ForestEvent>` 流式事件
- `maxRetries()` 失败重试次数（默认 0）

### 7.1.2 注册机制

`LeafExecutorRegistry` 通过 Spring 依赖注入自动收集所有 `LeafExecutor` 实现：

```java
// 各 executor 通过 @Component 注册，Spring 自动注入到 registry 的 List 中
// ForestExecutor 根据 node.nodeType() 匹配到对应的 executor
LeafExecutor executor = registry.find(node.nodeType());
```

## 7.2 各类执行器

### 7.2.1 AgentLeafExecutor
- `supportedType()` → `"task"`
- 核心执行器，完整的 ReAct 循环
- 支持工具调用、A2A 通信
- 详细实现见第六章

### 7.2.2 ChatLeafExecutor
- `supportedType()` → `"message"`
- 简单对话执行器，无工具调用
- 构建消息列表后直接调用 `model.chat()`
- 使用 `prebuiltMessages` 作为上下文，或使用 `PromptAssembler` 装配

```java
public Flux<ForestEvent> execute(TreeNode node, ExecutionContext ctx, EventSink sink) {
    List<Msg> messages = buildMessages(node);  // 从 prebuiltMessages 或 PromptAssembler
    return model.chat(messages, List.of(), ModelParams.defaults())
        .flatMap(response -> {
            sink.emit(ForestEvent.thinking(...));
            return Flux.just(ForestEvent.detail(...));
        });
}
```

### 7.2.3 SynthesisLeafExecutor
- `supportedType()` → `"synthesis"`
- 汇总多个子节点的输出
- 在 PARALLEL/CONSENSUS 模式中作为父节点执行器
- 合并所有子节点的 accumulatedContext，调用 LLM 做最终合成

### 7.2.4 SearchKBLeafExecutor
- `supportedType()` → `"kb_search"`
- 执行知识库检索
- 读取 metadata 中的 `kbName`、`topK`
- 调用 `KbSearchService` 执行向量检索
- 结果写入节点的 content 字段

### 7.2.5 ImageGenLeafExecutor
- `supportedType()` → `"image_gen"`
- 调用图像生成服务
- 将生成的图片信息写入 content

### 7.2.6 VideoGenLeafExecutor
- `supportedType()` → `"video_gen"`
- 调用视频生成服务

### 7.2.7 SpeechRecogLeafExecutor
- `supportedType()` → `"speech_recog"`
- 语音识别（ASR）
- 将音频转写为文本

### 7.2.8 SpeechSynthLeafExecutor
- `supportedType()` → `"speech_synth"`
- 语音合成（TTS）
- 将文本转为音频

### 7.2.9 ImageAnalysisLeafExecutor
- `supportedType()` → `"image_analysis"`
- 图像分析（多模态）
- 调用具备视觉能力的模型分析图片

## 7.3 扩展新的叶子节点

添加一个新叶子节点类型需要改动的文件：

| 步骤 | 文件 | 改动 |
|------|------|------|
| 1 | `ForestRepositoryAdapter.createNode()` | 添加新的 case 分支 |
| 2 | 新建 `/agent/forest/XxxExecutor.java` | 实现 LeafExecutor 接口 |
| 3 | `ForestNodeJpaRepository` | 无需改动（通用查询已覆盖） |
| 4 | 前端 `ForestTree.vue` | 如果新增节点需要特殊展示 |

这意味着：**添加一个新类型的叶子节点只需要创建 1 个执行器类 + 在 1 个 switch 中添加 1 个分支**。

## 7.4 执行器选择流程

```
ForestExecutor 遍历执行树
  → 对每个叶子节点，获取 node.nodeType()
    → 从 LeafExecutorRegistry 查找 matchingExecutor
      → 找到 → execute()
        → 失败 → 判断 maxRetries → 重试或报错
      → 没找到 → 抛出 IllegalArgumentException
```

## 关键设计决策

- **为什么不用策略模式（Strategy Pattern）？** LeafExecutor 本身就是策略模式——`supportedType()` 是策略标识，`execute()` 是策略实现。Spring 的 `@Component` 自动注册简化了策略的维护。
- **为什么不支持自定义节点类型？** 目前所有节点类型在 switch 中静态注册。如果需要动态扩展，可以在 `createNode` 中增加反射或 SPI 机制，但当前场景不需要。
