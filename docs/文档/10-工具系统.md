# 第十章：工具系统

## 核心问题

LLM 需要调用外部工具（读文件、执行命令、访问知识库、调用第三方 API）来完成任务。系统如何统一管理这些工具？工具的安全边界在哪里？

## 10.1 双轨工具架构

### 10.1.1 内部工具

内置在 Agent 进程中的工具，部署即用：

| 工具 | 注册类 | 功能 | 安全控制 |
|------|--------|------|---------|
| `file_list` | `FileToolsRegistrar` | 列出目录内容 | 沙箱路径限制 |
| `file_read` | `FileToolsRegistrar` | 读取文件内容 | 沙箱路径 + 文件类型限制 |
| `file_write` | `FileToolsRegistrar` | 写入文件 | 沙箱路径 + 写保护 |
| `file_search` | `FileToolsRegistrar` | 搜索文件内容 | 沙箱路径 |
| `bash` | `BashToolsRegistrar` | 执行 Shell 命令 | ShellPolicy 策略控制 |
| `send_agent_message` | `AgentLeafExecutor` | Agent 间通信 | 仅限同一 Forest |

### 10.1.2 外部工具（MCP）

通过 MCP（Model Context Protocol）服务器动态注册的工具：

```java
McpClientBuilder  // 构建 MCP 客户端
  → 连接到 MCP 服务器（Stdio / HTTP）
    → 获取服务器声明的工具列表
      → 注册到 ToolRegistry
        → 供 Agent 调用
```

MCP 服务器在 `sivan-agent/mcp/` 下配置，通过 `application.yml` 指定服务器地址和认证信息。

## 10.2 ToolRouter

### 10.2.1 职责

为给定任务节点解析可用工具列表：

```java
List<ToolSpec> tools = toolRouter.resolve(adapter, taskContent, execCtx);
```

- 输入：节点适配器（包含 agentName、serverId）、任务内容、执行上下文
- 输出：ToolSpec 列表（工具名 + 描述 + 参数 Schema）

### 10.2.2 路由逻辑

1. 如果有 `serverId`，加载该 MCP 服务器的工具
2. 如果有 `agentName`，加载 Agent 绑定的工具
3. 否则加载默认工具集（内部工具）
4. 如果 ToolRouter 返回空，从 `preferredToolSpecs` metadata 降级

### 10.2.3 ToolSpec 结构

```java
public record ToolSpec(
    String name,                    // 工具名（LLM 调用时使用）
    String description,             // 描述（LLM 理解工具用途）
    Map<String, Object> inputSchema // JSON Schema 参数定义
) {}
```

## 10.3 工具注册与执行

### 10.3.1 ToolRegistry

```java
public interface ToolRegistry {
    void register(ToolSpec spec, ToolExecutor executor);
    ToolExecutor find(String name);
    List<ToolSpec> list();
}
```

- Spring 启动时，`FileToolsRegistrar` 和 `BashToolsRegistrar` 注册内部工具
- MCP 服务器连接后，工具动态注册到 registry
- 去重：同名工具以先注册的为准

### 10.3.2 工具执行链路

```java
// AgentLeafExecutor.handleToolCalls()
Flux<ForestEvent> callEvents = toolCalls.stream()
    .map(tc -> ForestEvent.toolCall(...));

List<Mono<ToolCallResult>> monos = toolCalls.stream()
    .map(tc -> executeSingleTool(tc, accountId, node, convId))
    .toList();

Flux<ForestEvent> resultEvents = Flux.mergeSequential(monos)
    .flatMap(r -> ForestEvent.toolResult(...));
```

## 10.4 安全沙箱

### 10.4.1 文件沙箱

内部文件工具受 `_fileRootPath` 限制：

- `file_list` 和 `file_read` 只能访问 `_fileRootPath` 目录下的文件
- `file_write` 同样受限，不能写入系统目录
- 归档模式（`_archived`）：只读，禁止写入

### 10.4.2 命令沙箱

Bash 工具受 `ShellPolicy` 限制：
- 禁止的危险命令：rm -rf、mkfs、dd 等
- 超时控制：所有命令 30s 超时
- 工作目录限制：只能在 `_fileRootPath` 下执行

### 10.4.3 MCP 工具沙箱

MCP 工具的安全由 MCP 服务器自身负责：
- 系统不假设 MCP 工具的安全性
- 工具执行结果截断（3000 字符）
- 超时控制（5 分钟）

## 10.5 工具使用统计

`IToolUsageRepository` 记录每次工具调用：
- `account_id`、`agent_name`、`tool_name`、`server_id`
- `success`（是否成功）、`duration_ms`（耗时）
- `conversation_id`（关联对话）

这些数据用于：
- 路由系统的 Beta 参数更新
- 工具使用的热力图分析
- 异常工具调用的告警

## 关键设计决策

- **为什么双轨？** 内部工具保证核心功能可用（不依赖外部服务），MCP 工具提供扩展性。降低核心功能的延迟和依赖风险。
- **为什么不统一用 MCP？** 内部工具性能更好（进程内调用，零序列化开销），且不依赖 MCP 服务器的可用性。
- **文件工具为什么走 _fileRootPath 注入而不是全局配置？** 多项目场景下，每个执行树有自己的工作目录，注入方式支持按项目隔离。
