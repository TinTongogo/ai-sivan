# 第五章：编排策略

## 核心问题

当需要多个 Agent 协作完成一个复杂任务时，如何组织它们的执行顺序和数据流？

## 5.1 五种编排模式

### 5.1.1 SEQUENTIAL（顺序模式）
- 逻辑：子节点按 `sort_order` 依次执行，前一个的输出作为后一个的输入
- 数据流：`accumulatedContext` 元数据累积
- 实现：`SequentialModeStrategy`
- 适用：流水线任务（查资料 → 分析 → 写报告）

### 5.1.2 PARALLEL（并发模式）
- 逻辑：所有子节点同时执行
- 数据流：各节点独立输入，最终汇总到 SynthesisNode
- 实现：`ParallelModeStrategy`，通过 `Flux.merge()` 并发
- 适用：多角度分析（技术角度 + 商业角度 + 法律角度）

### 5.1.3 CONDITIONAL（条件模式）
- 逻辑：根据条件选择执行路径
- 数据流：只有满足条件的分支执行
- 实现：`ConditionalModeStrategy`
- 适用：if-else 分支逻辑

### 5.1.4 CONSENSUS（共识模式）
- 逻辑：所有子节点独立执行，结果通过 SynthesisNode 投票/合并
- 数据流：各节点独立，最终比较合并
- 实现：`ConsensusModeStrategy`
- 适用：需要多方验证的场景

### 5.1.5 HIERARCHICAL（层级模式）
- 逻辑：子节点可以继续是 InnerGoalNode（嵌套策略）
- 数据流：递归降级，每层有自己的编排策略
- 实现：`HierarchicalModeStrategy`
- 适用：复杂项目分解（项目 → 阶段 → 任务）

## 5.2 TreeMatcher 匹配机制

### 5.2.1 LlmTreeMatcher
- 使用 LLM 判断用户意图匹配哪种编排模式
- 将扁平的任务列表构建为层次化的 InnerGoalNode 树
- 提示词工程：`MatcherPrompts` 中的结构化 prompt

### 5.2.2 匹配回退
- LLM 匹配失败 → 默认 SEQUENTIAL
- 单任务 → 不构建内层节点，直接执行

## 5.3 节点间通信（A2A）

### 5.3.1 AgentMessageBus
- 基于 Reactor 的 `Sinks.Many` 实现
- 每个 Agent 订阅一个 topic（由其 nodeId 标识）
- 广播 topic：`broadcast`（所有 Agent 可订阅）

### 5.3.2 消息类型
- `REQUEST`：向特定 Agent 请求数据
- `RESPONSE`：回复请求
- `BROADCAST`：广播消息
- `DELEGATE`：委托任务

### 5.3.3 A2A 工具
- `send_agent_message`：LLM 通过此工具向其他 Agent 发送消息
- 参数：targetAgentId、content、type
- 消息持久化到 `forest_agent_messages` 表

## 5.4 SynthesisNode 的合成

### 5.4.1 合成时机
- 所有子节点执行完毕后触发
- SynthesisNode 本身也是一个可执行的叶子节点

### 5.4.2 合成策略
- SEQUENTIAL：汇总所有子节点的 `accumulatedContext`
- PARALLEL/CONSENSUS：合并所有子节点的输出，去重后合成
- 由 LLM 执行最终的合成提示词

## 5.5 执行树示例

```
用户：分析这个项目的技术栈、商业前景和风险

TaskNode: "分析项目"
└── InnerGoalNode (PARALLEL)
    ├── InnerGoalNode (SEQUENTIAL)
    │   ├── TaskNode: "读取项目文档"
    │   └── TaskNode: "分析技术栈" ← Agent A
    ├── InnerGoalNode (SEQUENTIAL)
    │   ├── TaskNode: "搜索市场数据"
    │   └── TaskNode: "评估商业前景" ← Agent B
    └── TaskNode: "分析法律风险" ← Agent C
        ↑ A2A: 向 Agent B 获取市场数据

执行完成后：
  SynthesisNode: "汇总技术栈+商业+风险分析"
```

## 5.6 与路由系统的交互

```
请求 → 路由系统确定 Agent → 编排策略构建执行树
  → 路由结果注入每个 TaskNode 的 metadata(agentName)
    → LeafExecutor 根据 agentName 加载对应的 Agent 提示词
      → 执行完成后 → 路由反馈更新 Beta 参数
```

## 关键设计决策

- **为什么不用 DAG（有向无环图）？** 树结构更简单，递归遍历自然支持深度优先/广度优先；DAG 需要拓扑排序和环检测，对 LLM 生成不可靠
- **为什么不支持图结构？** 99% 的协作场景可以用树 + A2A 通信覆盖，图的额外复杂性不值得
