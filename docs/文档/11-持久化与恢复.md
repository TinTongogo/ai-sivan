# 第十一章：持久化与恢复

## 核心问题

执行树在运行过程中可能因系统崩溃、网络中断、部署更新而中断。如何保证执行状态不丢失？中断后如何恢复？

## 11.1 数据库设计

### 11.1.1 forest_nodes 统一表

```sql
CREATE TABLE forest_nodes (
    node_id         VARCHAR(36)   PRIMARY KEY,
    forest_id       UUID          NOT NULL,
    node_type       VARCHAR(16)   NOT NULL,  -- 鉴别器
    parent_node_id  VARCHAR(36),              -- 自引用 FK
    sort_order      INT           NOT NULL,
    role            VARCHAR(16),              -- 消息角色
    mode            VARCHAR(16),              -- 编排模式
    status          VARCHAR(16),              -- 节点状态
    content         TEXT,
    metadata        JSONB,
    importance      DOUBLE PRECISION,
    estimate_tokens BIGINT,
    ...
    level           VARCHAR(16),              -- 记忆层级（V30 新增）
    archived        BOOLEAN,                  -- 归档标记
    important       BOOLEAN,                  -- 重要标记
    scope_id        UUID,
    summary         TEXT,
    ...
    vector          vector(1024),             -- 记忆向量
    created_at      TIMESTAMPTZ,
    updated_at      TIMESTAMPTZ   NOT NULL
);
```

### 11.1.2 索引策略

| 索引 | 类型 | 用途 |
|------|------|------|
| `(forest_id, parent_node_id)` | B-tree | 递归 CTE 树加载 |
| `(forest_id, node_type, sort_order)` | B-tree | 按类型查询 + 排序 |
| `(node_type, vector)` HNSW | 向量索引 | 记忆语义搜索 |
| `(account_id, level) WHERE node_type='memory'` | B-tree 部分 | 按层级过滤 |
| `(account_id, archived) WHERE node_type='memory'` | B-tree 部分 | 归档过滤 |
| `(retention) WHERE node_type='memory' AND NOT archived` | B-tree 部分 | 遗忘曲线调度 |

## 11.2 递归 CTE 树加载

### 11.2.1 查询

```sql
WITH RECURSIVE subtree AS (
    SELECT *, 0 AS depth FROM forest_nodes
    WHERE node_id = :rootNodeId AND forest_id = :forestId
    UNION ALL
    SELECT fn.*, st.depth + 1 FROM forest_nodes fn
    INNER JOIN subtree st ON fn.parent_node_id = st.node_id
    WHERE fn.forest_id = :forestId AND st.depth < 1000
)
SELECT * FROM subtree ORDER BY sort_order;
```

### 11.2.2 树组装

`ForestRepositoryAdapter.assembleTree()` 两遍扫描：
1. 第一遍：遍历行列表，创建每个节点的 Java 对象，按 node_id 建立索引
2. 第二遍：遍历父子关系，连接树结构

### 11.2.3 性能考虑

- 递归深度限制：1000 层（实际对话一般在 10 层以内）
- 单棵子树加载时间：< 10ms（100 个节点以内）
- 批量保存使用 `saveAll()` + `flush()`，减少数据库往返

## 11.3 序列化与反序列化

### 11.3.1 TreeNode → ForestNodeEntity（toEntity）

```java
// 核心流程
builder.nodeId(node.nodeId())
    .forestId(forestId)
    .nodeType(node.nodeType())
    .parentNodeId(resolveParentNodeId(node, forestId))
    .sortOrder(node.order())
    .mode(node.mode().name())
    .status(node.status().name());

// ContentNode → content + metadata + contentHash
if (node instanceof ContentNode cn) {
    builder.content(cn.content());
    builder.metadata(serializeMetadata(cn.metadata()));  // 过滤运行时 key
    builder.contentHash(computeContentHash(cn.content()));
    // estimateTokens ≈ content.length × 1.5
}
```

### 11.3.2 ForestNodeEntity → TreeNode（createNode）

按 `node_type` 分派到对应的实现类构造函数：
- `"task"` → `TaskNode(nodeId, content, status)` + metadata
- `"inner_goal"` → `InnerGoalNode(nodeId, mode, children, status)`
- `"message"` → `MessageNode(nodeId, content, role)` + metadata
- `"memory"` → `MemoryNode(nodeId, content, retention)` + metadata + vector

### 11.3.3 metadata 序列化控制

**运行时 key 黑名单**（Phase 1 治理的成果）：
```java
private static final Set<String> RUNTIME_METADATA_KEYS = Set.of(
    "prebuiltMessages", "preferredToolSpecs", "_forestId", "_accountId",

    "agentName", "serverId"

```

这些 key 在 `toEntity()` 序列化时被过滤，不写入 JSONB。

## 11.4 断点恢复

### 11.4.1 RecoveryManager

系统重启后，`RecoveryManager` 扫描 `status='RUNNING'` 的根节点：

```java
public void recoverOnStartup() {
    var roots = forestRepository.findAllRootNodesByStatus(NodeStatus.RUNNING);
    for (var root : roots) {
        var subtree = forestRepository.findSubtree(root.nodeId(), ...);
        var execCtx = ExecutionContext.create(...);
        resumeExecution(subtree, execCtx);
    }
}
```

### 11.4.2 恢复策略

1. **找到断点**：遍历子树，找到第一个 `status='PENDING'` 的节点
2. **重试前提**：如果前一个兄弟节点是 `FAILED`，恢复时跳过（不再重试）
3. **新上下文**：重新构建 ExecutionContext（accountId、projectId、conversationId 不变）
4. **继续执行**：从断点节点调用 `LeafExecutor.execute()`

### 11.4.3 恢复的局限性

- 运行时数据（`prebuiltMessages`、`agentName` 等）不在数据库中 → 恢复后需要重建
- 对简单对话，丢失运行时上下文不影响（AgentLeafExecutor 会降级为默认行为）
- 对复杂的多 Agent 编排，恢复后可能丢失部分 A2A 消息

## 11.5 事务管理

### 11.5.1 写策略

- `saveTree()`：在一个事务中，先用 `deleteChildren` 清除旧子树，再用 `saveAll` 批量写入
- `saveNode()`：单节点写入，立即 `flush()`
- `updateNodeStatus()`：节点粒度更新，不锁整棵树

### 11.5.2 外键约束

| 表 | 外键 | 状态 |
|----|------|------|
| forest_nodes (parent_node_id) | → forest_nodes (node_id) | 活跃 |
| forest_nodes (forest_id) | → forests (forest_id) | V29 已删除 |
| forest_hitl_reviews | → forests | V31 已删除 |
| forest_execution_logs | → forests | V31 已删除 |
| forest_agent_messages | → forests | V31 已删除 |
