# 第四章：组合式路由系统

## 核心问题

系统如何决定用户的请求应该由哪个智能体（Agent）处理？同一个请求是否需要绑定多个技能（Skill）？

## 4.1 设计哲学

### 4.1.1 为什么"组合式"？

传统路由系统将智能体绑定到一组固定的技能上。但实际场景中：
- 用户问"帮我查一下 Java 反射的用法，然后用 Python 写个示例"→ 需要 Agent A（Java）的领域知识 + Agent B（Python）的技能
- **智能体和技能是正交的**：Agent 是谁不重要，关键是它掌握什么能力

组合式路由的核心思想：
- **Agent 路由**：按任务内容匹配合适的智能体（谁来做）
- **技能匹配**：按任务内容匹配独立的技能片段（需要什么能力）
- 两者独立匹配，组合使用

## 4.2 Agent 三层贝叶斯路由

### 4.2.1 Tier 0：精确缓存
- 特征哈希（`featureHash`）精确命中
- 时间复杂度 O(1)
- 使用 `account_beta_params` 表的 (account_id, feature_hash, agent_name) 唯一约束
- 命中后按贝叶斯期望值排序

### 4.2.2 Tier 1：语义检索 + 贝叶斯加权
- 对 `routing_decisions` 表做 pgvector 余弦相似度搜索（HNSW 索引）
- 返回 Top-K 条历史决策
- 按 `Beta(alpha, beta)` 分布计算每个候选的 Thompson 采样分数
- 置信度 = α / (α + β)

### 4.2.3 Tier 2：Thompson 采样探索
- 从 `account_beta_params` 读取所有候选 Agent
- 对每个 Agent 从 `Beta(alpha, beta)` 采样
- 选择采样值最高的 Agent
- 探索-利用平衡：即使历史表现差，只要采样值高就有可能被选中

### 4.2.4 Tier 3：LLM 创建新 Agent
- 将任务内容发给 LLM，生成 Agent 定义（名称 + 描述 + 系统提示词）
- 自动注册到 `agents` 表
- 容错：如果 LLM 输出不合法，会尝试语义匹配已有 Agent

```
请求到达 → classifyInputType()
  ├── 非 task → 直接返回（不路由）
  └── task → Tier 0 精确缓存
      ├── 命中 → Tier 0 结果 + 技能匹配 → 返回
      └── 未命中 → Tier 1 语义检索
          ├── 置信度 > 0.7 → Tier 1 结果 + 技能匹配 → 返回
          └── 置信度 ≤ 0.7 → Tier 2 Thompson 采样
              ├── 采样有候选 → Tier 2 结果 + 技能匹配 → 返回
              └── 无候选 → Tier 3 LLM 创建
```

### 4.2.5 路由反馈闭环

`ForestExecutor` 在节点执行完毕后调用 `RouteFeedbackHandler.onNodeCompleted()`：
- 计算 `taskContent` 的特征哈希（`TaskFeatures.fromContent()`）
- 调用 `betaRepo.upsert()` 更新 Beta 参数
- 成功：alpha += 1；失败：beta += 1

**关键教训**：`IntentClassifier` 曾阻断这个闭环。当 `classifyInputType` 返回 "chat" 时，PgRouteEngine 直接返回 `agentName=null`，导致 `ForestExecutor` 的条件检查 `agentName != null` 不满足，`onNodeCompleted` 永不触发。修复方案：意图分类只影响前端展示，不影响路由决策。

## 4.3 技能独立匹配

### 4.3.1 技能模型
- `Skill` 定义在 `sivan-domain/agent`，包含 name、tags、content（提示词模板）、type（PERSONA/FUNCTION）
- 技能不绑定到 Agent，独立存储

### 4.3.2 匹配管道
- Tier 1：语义匹配（pgvector 检索 skill embeddings）
- Tier 2：探索（随机从低频使用的技能中选择）
- Tier 3：LLM 创建新技能
- 阈值：`SKILL_MATCH_THRESHOLD = 0.5`

### 4.3.3 组合结果
```java
RouteResult(agentName, category, tier, confidence, intent, matchedSkillIds)
```
- 前端展示：Agent 徽章 + 路由层级标签 + 置信度
- 执行阶段：Agent 的 systemPrompt + 技能的 content 合并为完整提示词

## 4.4 PgRouteEngine 实现细节

### 4.4.1 embedding 计算
- 使用 `TaskFeatures.fromContent()` 结构化提取特征
- 特征字符串 MD5 哈希作为 featureHash
- embedding 通过 `IEmbeddingService.embed()` 计算（Qwen3-Embedding-2B）

### 4.4.2 路由决策持久化
- `routing_decisions` 表：存储每次决策的 embedding、结果、成功状态
- pgvector HNSW 索引加速语义检索
- `account_beta_params` 表：存储 Beta 分布参数

### 4.4.3 性能调优
- Tier 0/1/2 总耗时 < 50ms（纯 SQL + 内存计算）
- Tier 3 耗时 1-3s（LLM 调用）
- `resolve()` 方法返回 `Mono<RouteResult>`，异步执行

## 4.5 与执行树的交互

```
路由结果 RouteResult
  └── agentName → addRuntimeMetadata(agentName)
      └── ForestExecutor.onNodeCompleted()
          └── RouteFeedbackHandler.onNodeCompleted()
              └── betaRepo.upsert() ← 路由反馈闭环
```

## 关键设计决策

- **为什么不用纯语义路由？** 贝叶斯路由能利用历史反馈自动调整，语义路由无法区分"好"和"坏"的匹配
- **为什么保留 Tier 0？** 精确匹配 O(1)，高频请求不需要每次都做向量检索
- **为什么 IntentClassifier 不阻断 agentName？** 见 4.2.5 的修复历程
