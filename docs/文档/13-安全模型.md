# 第十三章：安全模型

## 核心问题

AI 智能体能够读写文件、执行命令、调用外部 API。如何保证这些能力不被滥用？如何确保用户数据不被泄露？

## 13.1 认证与授权

### 13.1.1 JWT 认证

- 基于 JJWT 库（0.12.6）
- Token 有效期：24 小时
- 载荷包含：`accountId`、`role`、`projectIds`
- 每次请求通过 `SecurityContext` 解析当前用户

### 13.1.2 数据隔离

所有仓储接口显式携带 `accountId`：

```java
// 每个方法都有 accountId 参数
Forest findForestById(UUID forestId, UUID accountId);
void saveNode(TreeNode node, UUID forestId, UUID accountId);
List<Forest> listByAccountId(UUID accountId);
```

**规则**：禁止 ThreadLocal 隐式传递 accountId，防止跨账户数据泄露。

## 13.2 三层沙箱

### 13.2.1 OS 隔离

- Bash 工具执行环境：受限 Shell（`rbash`）
- 超时控制：所有命令 30s 超时
- 禁止的危险命令：rm -rf、mkfs、dd、:(){ :|:& };:（fork 炸弹）等
- 工作目录限制：只能在项目根目录下执行

### 13.2.2 文件沙箱

- 所有文件操作限制在 `_fileRootPath` 目录内
- `file_write` 只能写 `_fileRootPath` 下的文件
- `file_read` 只能读 `_fileRootPath` 下的文件
- `file_list` 只能列出 `_fileRootPath` 下的内容
- 归档模式（`_archived`）下禁止写入

### 13.2.3 归档只读

- 已归档/已完成对话的文件路径自动设为只读
- 通过 metadata 中的 `_archived` 标记传递
- 写入工具在收到 `_archived=true` 时拒绝执行

## 13.3 审计日志

### 13.3.1 审计范围

| 操作类型 | 记录内容 |
|---------|---------|
| 文件读取 | 文件路径、操作时间、账户 ID |
| 文件写入 | 文件路径、写入内容哈希 |
| 命令执行 | 命令内容、工作目录 |
| HTTP 请求 | URL、方法 |
| MCP 工具调用 | 工具名、参数 |

### 13.3.2 审计日志存储

所有审计事件记录到 `audit_logs` 表：
- `action_type`：操作类型
- `actor_id`：操作者（agentName）
- `account_id`：所属账户
- `resource`：操作对象
- `detail`：JSONB 详情
- `created_at`：操作时间

## 13.4 速率限制

| 限制项 | 阈值 | 周期 |
|-------|------|------|
| 每分钟请求数 | 60 | 1 分钟 |
| 每小时 Token 消耗 | 500K | 1 小时 |
| 每日 Agent 创建数 | 50 | 1 天 |
| 并发执行树数 | 10 | 同时 |

## 13.5 CORS 配置

- 前端开发环境（localhost:5173）允许跨域
- 生产环境限制到指定域名
- WebSocket/SSE 连接单独配置 CORS

## 13.6 安全最佳实践

### 13.6.1 输入过滤

- 用户输入在传递给 LLM 前进行长度校验
- 文件路径禁止包含 `..` 路径穿越
- 命令内容进行危险命令校验

### 13.6.2 数据加密

- JWT Token 使用 HMAC-SHA256 签名
- 敏感配置（API Key、密码）使用 AES-256-GCM 加密存储
- 数据库连接使用 TLS

### 13.6.3 输出过滤

LLM 返回的内容在推送到前端前进行：
- 敏感信息过滤（手机号、身份证正则匹配）
- 内容长度限制（防止拒绝服务）
- SQL 注入字符转义

## 13.7 与工具系统的交互

```
Agent 工具调用
  → 工具注册器安全检查（_fileRootPath、_archived）
    → 沙箱权限校验（ShellPolicy、FilePolicy）
      → 审计日志记录
        → 工具执行
          → 输出截断 + 敏感信息过滤
            → 返回结果给 Agent
```

## 关键设计决策

- **为什么三层沙箱？** 每一层防御不同的攻击面——OS 层防系统破坏，文件层防数据泄露，归档层防意外修改。单层防御不够。
- **为什么不用容器隔离？** 容器增加了部署复杂度。对于个人 AI 助手场景，三层软件沙箱足够。如果需要多租户隔离，可以再加容器层。
- **为什么显式传递 accountId？** 避免"忘记检查"的安全漏洞。如果 accountId 在线程上下文中隐式传递，可能某天新增一个服务时忘记设置，导致跨账户数据访问。
