# 第十四章：架构演进实录

## 核心价值

本章记录项目从 V1 到当前状态的所有重大架构决策——**哪些做对了、哪些做错了、为什么改**。这是其他技术书不会写的部分。

## 14.1 V1：目标管理系统

### 设计
- 独立的 `goals`、`goal_tasks`、`goal_events` 表
- 每个目标一个智能体，线性执行
- 无记忆、无路由、无编排

### 教训
- 太像传统项目管理工具，没有发挥 LLM 的优势
- 线性执行无法表达复杂协作
- 每个目标之间的上下文完全隔离

### 遗留痕迹
- `V5__drop_v1_goal_tables.sql` — 整个 V1 被删除
- 迁移 5 号文件就是这份"罪己诏"

## 14.2 V2：引入森林概念

### 设计
- `forests` 表作为顶级容器
- `forest_nodes` 表存储树结构
- 独立的 `messages`、`memory_entries` 表

### 做对了
- 树结构天然适合表达任务分解
- parent_node_id 自引用 + 递归 CTE ＝ 任意深度查询

### 做错了
- 分表过多：每新增一个节点类型就要新增一张表
- `forests` 表成为瓶颈：每次查询都要 JOIN forests

## 14.3 V3（当前）：统一森林

### 核心变更
- `V25__forest_nodes_unify.sql`：messages、memory_entries → forest_nodes
- `V28__forest_nodes_add_conversation_fields.sql`：forest_nodes 取代 forests
- `V29__drop_forests_fk.sql`：forests 表彻底脱钩

### 统一表的设计哲学
```
一张表（forest_nodes）+ 一个列（node_type）= 所有数据
```

### 带来的好处
- 递归 CTE 可以跨越消息、执行树、记忆
- 事务边界简化（一个 forest_id 搞定）
- 删除一个森林 = 删除该 forest_id 的所有节点

## 14.4 metadata 治理史（核心教训）

### 第一阶段：JSONB 是"万金油"
- "先用 JSONB 存着，后面再规范化"
- 结果：永远没有"后面"

### 第二阶段：垃圾场
- prebuiltMessages（全量 LLM 历史）→ JSONB
- prebuiltMessages 递归复制到每个子节点 → JSONB × N
- runtime key（_fileRootPath、_accountId）→ JSONB
- 过滤字段（level、archived、important）→ JSONB
- 查询时全部加载到 Java，filter 过滤 → 全表扫描

### 第三阶段：治理（本书撰写时的状态）
- Runtime key 黑名单 → persist 时过滤
- 过滤字段提升为列 + 部分索引
- 领域元数据保留在 JSONB（合法用途）

### 教训总结
```
不要相信"后面再规范化"——如果在设计时没有为某个字段安排列，
它就永远不会有列，永远会留在 JSONB 里。
```

## 14.5 拆表 vs 加列决策（ADR-2025-06）

### 背景
记忆查询在 Java 层对 metadata->>'level' 做内存过滤，需要修复。

### 方案 A：拆表
创建独立的 `memories` 表，从 forest_nodes 中拆出。

### 方案 B：加列
在 forest_nodes 上加专用列 + 部分索引。

### 决策过程
| 维度 | 拆表 | 加列 |
|------|------|------|
| 复杂度 | 高（双写、迁移、新实体） | 低（ALTER TABLE + 新列） |
| 架构一致性 | 破坏（memory 不属于森林） | 保持（"森林包含一切"） |
| 查询性能 | 最优（独立索引） | 接近最优（部分索引） |
| 恢复时间 | 长（数据迁移） | 短（无数据迁移） |

**最终选择加列**。核心考量：森林架构的理念就是"所有上下文在同一个容器中"，拆表在对抗架构。

## 14.6 linkMessageToTree 时序问题

### 问题
`linkMessageToTree()` 在 `messageRepository.save()` 之前调用。消息尚未写入 forest_nodes，`findById` 找不到，链接被静默丢弃。

### 为什么半年没发现
- 前端 PipelineDialog 的回退逻辑 `nodes.getFirst()` 总是"返回一个树"——虽然永远是第一个执行树
- "能用"不等于"正确"——在用户发现所有消息展开都显示同一棵树之前，这个 bug 潜伏了很久

### 修复
```
// 错误：
linkMessageToTree(msgId, treeId);  // msg 还没入库
messageRepository.save(msg);

// 正确：
messageRepository.save(msg);
linkMessageToTree(msgId, treeId);  // msg 已入库
```

### 教训
```
静默失败是最危险的 bug——没有报错、没有异常、只有错误的数据。
一切 findById 之后的 ifPresent 操作，都要检查 save 的时序。
```

## 14.7 IntentClassifier 阻断路由反馈

### 问题
新引入的 `IntentClassifier` 将请求分类为 "chat" 或 "task"。"chat" 请求直接返回 `agentName=null`，导致路由反馈永不触发，`account_beta_params` 全空。

### 根因
- 设计意图：chat 请求不需要路由
- 实际后果：chat 请求不产生路由反馈，需要路由反馈的 task 请求也受影响（因为同一个代码路径返回了 null）

### 修复
意图分类只影响前端展示（ChatLeafExecutor vs AgentLeafExecutor），不影响路由决策。`agentName` 始终由路由管道赋值。

### 教训
```
避免"早期返回"破坏系统闭环。新增的分支逻辑需要检查它是否
切断了某个已有的反馈环路。
```

## 14.8 仍在进行中的演进

### 服务拆分
- `ForestConversationService` 从 1000+ 行 → 3 个专注服务
- 趋势：按"链路"而不是按"层"组织代码

### 类型安全的 Metadata
- 当前：`Map<String, Object>` → 每次 get 都要 cast
- 目标：Java 21 sealed interface + record pattern matching

### 前端的执行树展示
- 当前：每次展开都请求 API，无缓存
- 目标：`Map<messageId, response>` 缓存 + WebSocket 推送增量更新

## 本书完结

架构没有终点，只有阶段性的平衡。Sivan 项目从 V1 的目标管理，到 V2 的森林雏形，再到 V3 的统一森林，每一步都是在"灵活"和"严谨"之间寻找平衡。

**metadata 治理、执行树链接、路由反馈闭环**——这三个问题贯穿了整本书，它们是架构设计中最容易被忽视、却又最难事后修复的细节。

希望本书能帮助你避免同样的坑。
