# 第二章：森林架构

## 核心问题

多智能体协作的核心矛盾在于：**每个智能体有自己的上下文，但协作需要共享上下文**。森林架构（Forest Architecture）的答案是：所有上下文都在同一个"森林"中，每个智能体看到的是森林中的一棵"树"。

## 2.1 设计哲学

### 2.1.1 为什么叫"森林"？
- **森林（Forest）** = 一个对话的全部上下文
- **树（Tree）** = 一次请求的执行路径
- **节点（Node）** = 一个具体的任务、消息、记忆
- 森林里可以有多个树（多轮问答），每棵树有自己的根，但它们共享同一片土壤（conversation container）

### 2.1.2 与 LangChain Agent 的区别
- LangChain：每个 Agent 独立，Agent 间通过 Tool 通信
- Sivan：所有 Agent 在同一个 Forest 中，Agent 间通过 A2A（Agent-to-Agent）消息总线通信
- 关键优势：编排策略可跨 Agent 调度，上下文共享在架构层面而非工具层面

## 2.2 ContextForest

### 2.2.1 领域模型
- `ContextForest` 是领域层的聚合根
- 包含：多个 `ContextTree`、`Epoch`（版本号）、`TopicNode`（话题结构）
- 每个 Forest 对应一次对话（conversationId = forestId）

### 2.2.2 预算分配（Scenes）
- 4 种场景：CHAT、SQUAD、KB、TOOL
- 每种场景有不同的 Token 预算分配策略
- `Scene` 决定 ContextForest 中每个 ContextTree 的最大长度

## 2.3 ContextTree

### 2.3.1 三棵树
- **KBTree**：知识库片段树，从向量检索结果构建
- **MemoryTree**：记忆片段树，从闪回扫描结果构建
- **ToolChainTree**：工具调用链树，从历史工具调用记录构建

### 2.3.2 树的构建时机
- KBTree：在每次用户输入后，通过 RAG 检索构建
- MemoryTree：在每次用户输入后，通过 FlashbackScanner 扫描构建
- ToolChainTree：在每次工具调用后构建

## 2.4 执行树（Execution Tree）

### 2.4.1 节点类型体系

| 节点类型 | Java 类 | 角色 |
|---------|---------|------|
| `inner_goal` | `InnerGoalNode` | 编排策略的内部节点，决定子节点的执行模式 |
| `task` | `TaskNode` | 叶子节点，由 AgentLeafExecutor 执行 |
| `synthesis` | `SynthesisNode` | 汇总节点，将多个子节点的输出合成 |
| `message` | `MessageNode` | 对话消息节点 |
| `memory` | `MemoryNode` | 记忆节点 |
| `file_snapshot` | `FileSnapshotNode` | 文件快照 |
| `context_block` | `ContextBlockNode` | 上下文块（RAG 结果、工具输出等） |
| `kb_search` | `SearchKBNode` | 知识库搜索节点 |

### 2.4.2 节点接口层次

```
TreeNode（基接口）
├── ContentNode（有内容 + 元数据）
│   ├── TaskNode（可执行 + 可压缩）
│   ├── SynthesisNode（可执行）
│   ├── MessageNode
│   ├── MemoryNode（可压缩）
│   ├── FileSnapshotNode
│   ├── ContextBlockNode
│   └── SearchKBNode（可执行）
├── ExecutableNode（可执行：有 mode + status）
│   ├── InnerGoalNode（可压缩）
│   ├── TaskNode
│   ├── SynthesisNode
│   └── SearchKBNode
└── CompressibleNode（可压缩：有 importance）
    ├── InnerGoalNode
    ├── TaskNode
    └── MemoryNode
```

### 2.4.3 树的执行流程
1. **构建**：`ForestExecutionOrchestrator.buildTree()` 创建 TaskNode 根节点
2. **编排**：`TreeMatcher.match()` 匹配编排模式，构建 InnerGoalNode 子层
3. **执行**：`ForestExecutor` 递归遍历树，为每个叶子节点分配 LeafExecutor
4. **完成**：更新节点状态，记录 Token 用量，触发路由反馈

## 2.5 Metadata 治理（重要设计决策）

### 2.5.1 为什么 metadata 变成了垃圾场
- 最初设计：metadata 是 JSONB 字段，用于存储"扩展属性"
- 实际使用：塞进了运行时上下文、大对象、过滤条件
- 后果：JSONB 膨胀、内存过滤、类型不安全

### 2.5.2 治理方案
- **Phase 1**：运行时 key 黑名单（`RUNTIME_METADATA_KEYS`），persist 时过滤
- **Phase 2**：记忆专用列（`level`/`archived`/`important`/`scope_id`/`summary`），SQL WHERE 替代内存过滤
- **未来**：typed metadata（Java 21 sealed interface + record pattern）

### 2.5.3 当前 metadata 的合法用途
- `filePath` — FileSnapshotNode 的文件路径
- `blockType` — ContextBlockNode 的块类型
- `kbName`、`topK` — SearchKBNode 的搜索配置
- `messageId`、`model`、`totalTokens` — MessageNode 的扩展信息

## 2.6 与相邻模块的交互

```
对话系统 ──→ 构建执行树 ──→ 森林架构
                                 │
                    ┌────────────┼────────────┐
                    ↓            ↓            ↓
              编排策略       执行器       持久化
               (TreeMatcher) (LeafExecutor) (forest_nodes)
                    │            │
                    ↓            ↓
              路由系统      工具系统
```
