# 第三章：消息与执行树的关联

## 核心问题

消息（MessageNode）和执行树（TaskNode/InnerGoalNode）都存储在 forest_nodes 表中。它们之间的关系是什么？为什么前端展开消息时经常显示错误的执行树？

## 3.1 树的拓扑结构

### 3.1.1 对话容器

每个对话有一个"对话容器节点"（`node_type='conversation'`，`node_id = forestId = conversationId`）：

```
conversation (node_type='conversation', node_id=convId)
├── user_msg_1 (node_type='message', parent=convId)
├── tree_1_root (node_type='task', parent=convId)
│   ├── inner_goal (node_type='inner_goal')
│   │   ├── task_A (node_type='task')
│   │   └── task_B (node_type='task')
│   └── assistant_msg_1 (node_type='message', parent=tree_1_root)
├── user_msg_2 (node_type='message', parent=convId)
├── tree_2_root (node_type='task', parent=convId)
│   └── assistant_msg_2 (node_type='message', parent=tree_2_root)
└── ...
```

### 3.1.2 默认 parent 设置

在 `ForestRepositoryAdapter.resolveParentNodeId()` 中：
- 树节点有 `parent()` → 使用父节点的 nodeId
- 树节点无 `parent()`（根节点）→ `parent = forestId.toString()`（挂到对话容器下）
- MessageNode → 同样通过此逻辑，默认挂到对话容器下

**问题**：消息默认是对话容器的直接子节点，不是执行树的子节点。

## 3.2 linkMessageToTree 机制

### 3.2.1 意图

`linkMessageToTree(msgId, treeNodeId)` 的作用是将消息节点的 `parent_node_id` 从 `conversationId` 改为执行树根节点的 ID。这样消息就"挂"到了执行树下面。

### 3.2.2 时序 bug（核心教训）

```java
// 错误版本（V29 之前的所有代码）：
linkMessageToTree(assistantMsg.getMessageId(), tree.nodeId());
// 此时 assistantMsg 刚在内存中创建，尚未写入 forest_nodes
// findById 返回 empty，ifPresent lambda 不执行
// 链接被静默丢弃

messageRepository.save(assistantMsg);
// 消息此时才写入 forest_nodes，但链接操作已经过去了
```

**为什么没有任何错误日志？** `linkMessageToTree` 内部：
```java
forestNodeJpaRepository.findById(messageId.toString()).ifPresent(entity -> {
    entity.setParentNodeId(treeRootNodeId);
    forestNodeJpaRepository.save(entity);
});
```
`findById` 返回 `Optional.empty()`，`ifPresent` 什么都不做——没有 `orElseThrow`、没有 `log.warn`。

**教训**："`findById` 后的 `ifPresent` + 静默失败"是最危险的 bug 模式。

### 3.2.3 修复

```java
// 正确版本：
messageRepository.save(assistantMsg);
// 消息已写入 forest_nodes，findById 能找到了
linkMessageToTree(assistantMsg.getMessageId(), tree.nodeId());
// 链接成功
```

同时增加了用户消息的链接：
```java
linkMessageToTree(userMessage.getMessageId(), tree.nodeId());
```

## 3.3 回退策略（Fallback）

### 3.3.1 为什么需要回退

- 旧数据（修复前保存的消息）没有正确的 parent_node_id
- 对话中、对话结束未生成执行树的消息也没有链接

### 3.3.2 回退逻辑（当前）

```java
// 1. 按 created_at 降序找到对话下所有执行树根节点
var treeRoots = findExecutionTreesByForest(conversationId);

// 2. 找到 created_at 最接近且不晚于消息创建时间的执行树
for (var tr : treeRoots) {
    if (!tr.getCreatedAt().isAfter(msgEntity.getCreatedAt())) {
        matchedRootId = tr.getNodeId();
        break;
    }
}
```

### 3.3.3 为什么不能总用 `nodes.getFirst()`

旧回退逻辑使用 `nodes.getFirst()`——永远返回对话中第一个执行树。导致：
- 第 1 轮问答 → 正确（第一个就是自己）
- 第 2 轮问答 → 错误（显示第 1 轮的树）
- 第 N 轮问答 → 永远显示第 1 轮的树

**业务影响**：用户看到的"执行进度"永远是第一次执行的进度。

## 3.4 前端的树展示

### 3.4.1 PipelineDialog

前端 `PipelineDialog.vue` 通过 API `/v2/conversations/{convId}/forest/{msgId}` 获取某条消息的执行树：

```
点击消息 → fetchMessageForest(convId, msgId)
  → getMessageForest()
    → 查 msg 的 parentNodeId
      → 有链接 → findSubtree() 加载执行树
      → 无链接 → 回退 → findSubtree(匹配的树根)
        → 无任何树 → buildSimpleChatTree()（单步树）
```

### 3.4.2 缓存策略

每次展开都请求 API 是浪费。当前实现使用 `Map<messageId, response>` 缓存，同一消息不重复请求。首次失败后 2s 自动重试一次。

### 3.4.3 ForestTree 组件

递归组件，接收 `ForestTreeNode` 接口：
- name：节点名称
- status：COMPLETED/RUNNING/PENDING/FAILED
- mode：编排模式（用于显示模式图标）
- agent：Agent 名称徽章
- children：子节点列表
- routeTier/routeConfidence：路由层级和置信度显示

## 3.5 与持久化层的关系

```
消息写入 → messageRepository.save()
  → ForestNodeJpaRepository 保存为 node_type='message'
    → linkMessageToTree 更新 parent_node_id
      → findSubtree 时通过递归 CTE 加载整棵树
```

**关键表结构**：

| 节点 | node_id | parent_node_id | 说明 |
|------|---------|---------------|------|
| 对话容器 | conversationId | NULL | 整个森林的根 |
| 执行树根 | treeRootUUID | conversationId | 一轮问答的执行树 |
| 用户消息 | msgUUID | treeRootUUID | 触发这轮问答的消息 |
| 助理消息 | msgUUID | treeRootUUID | 这轮问答的回复 |
